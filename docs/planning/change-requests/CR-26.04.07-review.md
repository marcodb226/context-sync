## Reviewer Notes

### Scope Fit

The amendment is correctly scoped as a material amendment under planning-model
§5: it adds a new named design ticket (M5-D1), changes the dependency version,
and changes the dependency list of an existing ticket (M5-1). The done-ticket
amendment rule is respected — M1-D2 is preserved as historical reference and
not retroactively expanded, and the new work is a follow-on design ticket that
produces its own standalone deliverable.

### Candidate Decisions

All five candidate decisions are well-reasoned:

- Staying on v1.0.0 is correctly rejected — writing raw-GQL helpers that the
  domain layer now handles would be wasted work.
- Patching M1-D2 in place is correctly rejected — rewriting a completed
  ticket's artifact to describe a different library version would violate the
  done-ticket rule.
- Producing a new standalone audit (M5-D1) rather than in-place patching is
  the right structural choice.

### Gap Closure Analysis

The five-gap table is thorough and its v1.1.0 status assessments are consistent
with the linear-client release notes. The per-issue vs. batched distinction is
correctly drawn for comment metadata and relation metadata.

### Application Pathway

The dependency chain M5-D1 -> M5-1 is sound for the gateway implementation
itself. The concern in CR-26.04.07-R1 is about the edges: v1.1.0 consequences
that affect code outside the gateway (renderer, types, error handling) need a
stated mechanism. The user's question — "how are the findings of that new
design task being applied?" — is valid. Some consequences are genuinely
unpredictable until M5-D1 completes; others (the breaking change to
`Issue.get_comments()` root-thread projection, `User.app` replacing
`is_app_actor()`) are already known from the CR's analysis and should be
traceable to specific tickets now.

| --- | --- | --- |

The recommended resolution for the five duplicates is re-export from upstream
(e.g. `from linear_client.types import IssueId as IssueId`), which preserves
the existing 11 import sites unchanged while unifying the `NewType` identity
across the dependency boundary. For `Timestamp` / `IsoTimestamp`, the audit
should evaluate whether to adopt the upstream name or keep an explicit alias.
The three context-sync-only types remain local since they have no upstream
counterpart.

#### Overall Assessment

The CR is well-structured and correctly applies the done-ticket amendment rule.
The gap-closure analysis is thorough, the candidate decisions are sound, and the
M5-D1 / M5-1 dependency chain is the right structural choice.

The most important gap is the type-surface reconciliation (R6 High, R9 Medium).
The upgrade creates a concrete Pyright type incompatibility across the gateway
boundary that the CR does not track. This is not a hypothetical future concern —
it is a known consequence of upgrading to a dependency version that exports the
same `NewType` aliases that context-sync independently defined. The CR should
ensure that M5-D1's adapter-boundary definition includes a type-mapping
deliverable and that M5-1 is explicitly scoped to use a single `NewType`
identity per domain concept.

The secondary weakness is that the CR treats its capability and breaking-change
tables as informational context without mapping their consequences to concrete
tasks or disposition categories. Pass 1 R2 flagged this at the table level;
pass 2 R5, R6, and R9 identify specific deliverables where that gap produces
tracked obligations the CR does not account for.
