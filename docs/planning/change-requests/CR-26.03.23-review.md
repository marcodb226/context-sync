## Reviewer Notes

### Scope Fit

The CR correctly identifies the three gaps between the current M4 boundary and a
credible 1.0.0: the missing real gateway, the missing supported docs surface,
and the missing release process. Each gap is grounded in specific review findings
(M4-1-R1, M4-2-R1, M4-2-R2, M4-2-R3), the README's pre-release notice, and
the relevant common-policy documents. The two-milestone split — gateway first,
then docs/release — respects the natural sequencing: you cannot document or
release a runtime that does not work.

### Candidate Decisions

All seven candidate decisions are well-reasoned. In particular:

- Dropping the "one broad post-M4 ticket" option in favor of a two-milestone
  split correctly recognizes that the missing scope spans independent
  deliverables with different sequencing and review needs.
- Dropping the "keep release workflow out-of-band" option correctly recognizes
  that the release process involves material product-delivery requirements
  (checklist, version-state guardrails, next-cycle bootstrap) that should be
  planned, not improvised.
- Dropping the "defer real gateway to future work" option correctly recognizes
  that a fake-gateway-only 1.0.0 is not credible.

### Sequencing and Dependencies

The dependency graph is sound:

- M5-1 depends on M1-D2 (adapter boundary), M3-O1 (comments_signature
  settlement), and M4-1 (CLI surface) — all Done. M5-1 can start immediately.
- M5-2 depends on M5-1 and M4-2 — correct, since public validation requires
  both the real gateway and the logging/validation infrastructure.
- M6-1 depends on M5-2, M4-R2, and M4-3 — correct, since docs must reflect
  the validated real runtime and the settled API surface. The transitive chain
  through M4-R2 to M4-R1 ensures the docs also wait for the CLI review
  outcome.
- M6-2 depends on M6-1 — correct, since the release process must validate the
  docs surface.
- M6-R1 depends on M6-1 and M6-2 — correct, since the readiness review
  evaluates the total shipped boundary.
- M6-3 depends on M6-O1 and M6-R1 — correct, since the release cut needs both
  publication prerequisites and readiness confirmation.

The remaining M4 tickets (M4-R1, M4-R2, M4-3) are correctly left unchanged,
and their outcomes flow into M6-1 through the existing dependency chain. The
CR's M6-1 detailed notes explicitly handle the case where M4-R2 produces
additional follow-on API changes beyond M4-3.

### Done-Ticket Amendment Rule

The CR correctly represents all new work as new follow-on tickets rather than
widening completed tickets. M5-1 cites M1-D2, M3-O1, and M4-1 as satisfied
prerequisites — appropriate since those tickets' accepted repository artifacts
already support the new dependency. M5-2 is the explicit defer destination for
M4-2-R3, correctly making that deferral relationship traceable.

### Missing Prerequisites

No missing prerequisites identified. M1-O1 (live Linear validation
environment) is already Done and implicitly available for M5 gateway
verification. M6-O1 correctly covers the release publication prerequisites
that are a human decision.

### Missing Design Items

No missing design tickets. The gateway adapter boundary is fully defined by
M1-D2. The release workflow framework exists in common policy. The docs
surface does not need a separate design document.

### Over- or Under-Commitment

The scope is proportional. Seven new named items (two milestones) to close
three identified gaps is appropriate. The individual ticket scopes are
well-bounded: M5-1 is constrained by the M1-D2 boundary, M5-2 is constrained
by the existing public surface, M6-1 is constrained by the shipped behavior,
M6-2 is constrained by the common release-workflow policy, and M6-3 is the
mechanical execution of M6-2's defined process.

### Backlog-Source Traceability

Good. Every proposed item traces to specific review findings (M4-1-R1,
M4-2-R1, M4-2-R2, M4-2-R3), the README pre-release notice, the active plan,
or the common policy documents. The CR's candidate-sources section is complete.

## Residual Risks

- The CR does not address whether `linear-client` v1.0.0 will prove sufficient
  for all M5-1 gateway operations. The M5-1 detailed notes acknowledge this
  risk ("If `linear-client` 1.0.0 proves insufficient...") but the plan-level
  risk register update in section 5 should also carry this risk explicitly,
  since a `linear-client` gap discovered during M5-1 could block the entire
  1.0.0 path.
- The M5-2 manual smoke recipe is the sole real-environment validation gate
  for the 1.0.0 release. If that recipe is not thorough enough, production
  users will be the first to exercise certain gateway paths against real
  Linear.

### Second-Pass Reviewer Notes

- The overall amendment direction is still strong. Adding a real gateway
  milestone and a separate docs/release milestone is the right way to extend
  the plan without silently widening completed work or mutating the remaining
  Milestone 4 tickets.
- The candidate-decisions table remains one of the draft's strengths. The
  keep/drop rationale is clear, especially around rejecting a single
  catch-all "ship 1.0.0" ticket and around preserving the done-ticket
  amendment rule.
- Changelog scope check: the draft does mention `CHANGELOG.md`, but only in
  [M6-2](CR-26.03.23.md#L191) and the Milestone 6 exit criteria as part of the
  first stable release bootstrap. That is consistent with
  [docs/policies/common/coding-guidelines.md](../../policies/common/coding-guidelines.md),
  which makes changelog maintenance optional before `1.0.0`. Stage 3 should
  preserve that "bootstrap for the first stable release" framing and should not
  broaden it into an expectation that pre-`1.0.0` milestones maintain a live
  changelog.
- I do think the active-plan framing edits should be read carefully during
  Stage 3 so the promoted plan no longer sounds like it ends at a bootstrap
  boundary once this amendment is applied. That is more of a consistency pass
  than a new formal finding because section 1 of the CR already points in that
  direction.
- This was planning-only review work, so no repository lint, format, or test
  commands were run.

### Second-Pass Residual Risks

- Even with CR-R3 fixed, the amendment still depends on a human choosing a real
  publication destination and making the required credentials or approval path
  available in time. That operational dependency is legitimate, but it remains
  a schedule risk until the destination is explicitly chosen.
- If the owner accepts CR-R4 only partially, Phase A/B execution for
  [M6-2](CR-26.03.23.md#L191) may need an extra round of policy interpretation
  to decide whether the repo is following the standard Python release-script
  layout or intentionally documenting an exception.
