## Gap Summary

The common-policy layer has added or materially updated several modules since the
active plan was last amended. This analysis classifies each gap as either:

- **Immediate** - not covered by any existing plan item and should be addressed
  before M5 work begins, because later milestones will build on the corrected
  baseline
- **Plan-covered** - already addressed by an existing M5 or M6 ticket
- **Advisory** - new policy artifact that does not create a compliance gap but
  changes how existing work is evaluated

## Proposed Amendment

If accepted, this amendment would add a new **Milestone 4.2** between
Milestone 4.1 and Milestone 5 with the scope limited to closing the seven
immediate compliance gaps identified above. This keeps M5 focused on the real
gateway and ensures the compliance baseline is in place before later milestones
build on it.

### Rationale for a separate milestone

- G-1 through G-4 affect the quality gate and type-checking baseline that every
  subsequent implementation ticket will run against. Landing them inside M5
  would mean the first M5 ticket must carry both gateway work and toolchain
  changes simultaneously.
- G-5 and G-6 are documentation/navigability artifacts that become more useful
  once the real gateway exists but whose structure should be established now
  while the public API surface is freshly settled after M4.1.
- G-7 is a README gap that blocks any new contributor from bootstrapping the
  common-policy layer.

### Candidate Decisions

| Candidate | Decision | Rationale |
| --- | --- | --- |
| Address all seven gaps as ad hoc work outside the plan | Drop | The gaps are material enough to warrant tracked execution artifacts; ad hoc work would leave no review trail |
| Fold the gaps into existing M5 tickets | Drop | M5 tickets are scoped to gateway and runtime validation; mixing toolchain/docs compliance work dilutes both scopes and makes Phase B review harder |
| Add the gaps as tickets inside Milestone 4.1 | Drop | M4.1 is already complete (both tickets at Phase C done); reopening it violates the done-ticket amendment rule |
| Create a new Milestone 4.2 for policy compliance | Keep | Clean milestone boundary, minimal disruption to M5/M6 scope, and all seven gaps get tracked execution artifacts with Phase B review |

#### Detailed Ticket Notes

##### M4.2-1 - Quality gate entry point and static analysis baseline

- `scripts/validate.sh` is the canonical "validate everything" command per
  [coding-guidelines.md](../../policies/common/coding-guidelines.md). It must
  run all declared quality gates and exit non-zero on any failure.
- Pyright config should target the same Python version as the rest of the
  toolchain (3.13). Use `pythonVersion = "3.13"` and `typeCheckingMode = "standard"`
  as the starting baseline; tighten to `strict` only if zero errors is achievable
  without excessive `type: ignore` annotations.
- Resolving Pyright errors may require adding or tightening type annotations in
  some modules. This is expected and should be treated as part of the ticket scope.
- The `py.typed` marker is a zero-byte file at `src/context_sync/py.typed`. The
  `[tool.setuptools.package-data]` entry ensures it ships in the wheel.
- Update `scripts/validate.sh` in lockstep with any future toolchain changes so
  the script remains the single source of truth for quality gates.

##### M4.2-2 - Coverage tooling and agent-awareness artifacts

- Coverage config should enable branch coverage and source from `src/`, omitting
  `version.py` per
  [python/agent-awareness.md](../../policies/common/python/agent-awareness.md).
- The `examples/` script must be runnable against the fake gateway since the real
  gateway is not yet available. Use the `_gateway_override` testing hook with a
  brief comment explaining that the example will switch to the real constructor
  once the gateway lands in M5.
- The dependency map should follow the minimum content requirements from
  [python/agent-awareness.md](../../policies/common/python/agent-awareness.md):
  main entry points, key internal modules, workflow-to-code map, anti-patterns,
  and async ownership. Keep to 1-2 pages.
- The README common-policy bootstrap section should cover: (1) that the project
  uses a shared common-policy layer, (2) system prerequisites (`ripgrep`, `cloc`),
  (3) exact symlink setup commands, (4) verification, (5) recovery, and
  (6) link to `docs/policies/common/README.md`.

#### Exit Criteria

1. `scripts/validate.sh` runs all quality gates (Ruff lint, Ruff format check,
   Pyright, pytest) and exits 0 on a clean tree.
2. Pyright reports zero errors across the source tree.
3. `py.typed` is present and included in the built wheel.
4. `pytest --cov --cov-report=term-missing` produces a branch coverage report.
5. `examples/` contains at least one runnable script that exercises the primary
   `ContextSync` workflow.
6. `docs/design/dependency-map.md` covers all minimum-content requirements.
7. README includes a self-contained common-policy bootstrap section.
