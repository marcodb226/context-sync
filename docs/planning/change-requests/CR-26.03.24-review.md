## Reviewer Notes

### Scope Fit

The amendment is correctly scoped. FW-10 was created explicitly to hold the CLI
simplification pending a plan amendment, M4-R1 and M4-R2 each concluded their
follow-on work needed to flow through the planning model, and all three cited
prerequisites (M4-3, M4-R1, M4-R2) are `Done`. Inserting M4.1 between M4 and
M5 so that M5 work starts from the settled four-command surface is the right
delivery shape.

### Candidate Decisions

All five candidate decisions are well-reasoned. In particular:

- Rejecting additional M4 tickets correctly respects the done-ticket amendment
  rule: silently widening `Done` tickets is prohibited, and inserting a clean
  new milestone keeps M4's closeout intact.
- Rejecting M5 absorption correctly recognizes that the CLI/library surface
  simplification and the real gateway wiring are independent deliverables with
  different review needs and different failure modes.
- Rejecting a two-ticket split is well-supported by the explicit
  [M4-R2 §3](../../execution/M4-R2.md) recommendation that the library changes
  should be implemented together with the CLI rename. Landing them atomically
  is the right call.
- Rejecting a separate milestone-level review ticket is sound: M4-R1 and M4-R2
  already constitute two independent review passes whose accepted artifacts
  fully prescribe the implementation. The normal Phase B review of M4.1-1
  provides the remaining independent check.

### Use-Case Coverage

The six-use-case mapping in §2.2 is a valuable addition to the plan. It makes
the standalone `sync` mode (use case 4) explicit and clearly distinguishes it
from `sync TICKET` (use cases 1 and 2). Standalone `sync` is not a named item
in M4-R1's definitive proposal, but it is the natural consequence of making
TICKET optional, and [M4-R2 §3](../../execution/M4-R2.md) explicitly defines
`sync(key: str | None = None)` with the same semantics. The traceability is
present even if not surfaced in one explicit sentence in the CLI review.

### Sequencing and Dependencies

The dependency graph is sound:

- M4.1-1 depends on M4-3, M4-R1, and M4-R2 — all `Done`. The ticket can start
  immediately.
- Adding M4.1-1 to M5-1's dependency list is correct. M5-1's gateway tests
  should route through the settled four-command surface rather than through a
  surface that will change before M5 is done.
- M5-2 and all Milestone 6 work continue to flow correctly downstream.

### Done-Ticket Amendment Rule

The CR complies. M4-3, M4-R1, and M4-R2 are cited as satisfied prerequisites
whose accepted repository artifacts already support the new dependency. No
completed ticket is being widened.

### Backlog-Source Traceability

Good. Every required change traces to M4-R1's definitive proposal, M4-R2's
§§3–4, the M4-R1-review Phase C findings, and FW-10's scope. The CR's
candidate-sources section is complete and cross-links the relevant execution
artifacts in both directions.

### Over- or Under-Commitment

The scope is proportional. One implementation ticket for a fully prescribed
surface change — with specific test requirements for each behavioral boundary —
is an appropriate commitment level. The ticket description is detailed enough
that execution can proceed from repository artifacts rather than
conversational memory.

## Residual Risks

- M4.1-1 touches the public CLI parser, three public library method signatures,
  `__all__`, user docs, and all tests that reference the old surface. The
  combined scope is wide. The most likely execution risk is that test updates
  are incomplete — specifically that some integration test still calls `add` or
  `remove-root` and is silently skipped rather than explicitly removed. The
  `__all__` test requirement and the parser-rejection tests in the ticket
  description guard the critical boundaries; execution should ensure test
  removal is auditable (deleted, not commented out).
- M5-1's dependency on M4.1-1 means that any unexpected complexity in the
  CLI simplification directly delays the real gateway work. If M4.1-1 surfaces
  a design ambiguity (for example, around the traversal-config preservation
  path for the standalone `sync` case with multiple roots), that could hold up
  M5-1. The exit criterion for traversal-config preservation (criterion 4) is
  precise enough to prevent scope creep, but the implementation should verify
  this edge case early.
