## Background

`linear-client` v1.1.0 has been released and is now available in the
multi-repo workspace. The release is a broad API-shape refinement that keeps
the two-layer architecture intact but makes the domain layer substantially
more complete and tightens several public contracts that were provisional in
v1.0.0. It also introduces intentional backward-incompatible changes
documented in its release notes.

The M1-D2 domain-coverage audit was written against v1.0.0. That ticket is
done and its artifact is preserved as historical reference (renamed to
`linear-domain-coverage-audit-v1.0.0.md`). This amendment adds a new design
ticket to produce a clean v1.1.0 audit and adapter-boundary definition, updates
the M5-1 ticket notes to reference the new audit instead of the superseded one,
and records the remaining gaps that still require raw-GraphQL helpers.

## v1.1.0 Gap Closure Analysis

The M1-D2 audit identified five categories of missing upstream capability in
v1.0.0. The table below records how v1.1.0 changes each:

| M1-D2 Gap | v1.0.0 Status | v1.1.0 Status | Impact on Adapter Boundary |
| --- | --- | --- | --- |
| **Traversal and rendered relations** â€” read-only issue-relation listing | Missing at both layers | **Closed** â€” `Issue.get_links()` returns `list[IssueLink]` with `link_type`, `updated_at`, `archived_at`; `IssueLink` is a full-CRUD domain object | Single-issue relation reads move to the domain layer. The `get_ticket_relations(issue_ids)` raw-GQL helper is only needed for batched multi-issue reads if the gateway batches relation fetches across the tracked set. |
| **Comment metadata for `comments_signature`** â€” `updatedAt`, parent/root topology, thread `resolved`, deletion signal | Missing at the domain layer | **Partially closed** â€” `Comment.updated_at`, `.resolved_at`, `.parent`, `.children`, `.is_resolved()`, `.is_placeholder()` are now domain-layer; however, no batch-by-issue-set comment-metadata endpoint exists | Per-issue comment metadata is domain-layer. The `get_refresh_comment_metadata(issue_ids)` raw-GQL helper is still needed for the batched refresh-cursor pass across the tracked reachable set. |
| **Relation metadata for `relations_signature`** â€” per-issue relation reads | Missing at both layers | **Partially closed** â€” `Issue.get_links()` provides per-issue relation metadata including `updated_at` and `archived_at`; no batch-by-issue-set relation-metadata endpoint exists | Same pattern as comments: per-issue reads are domain-layer, but the `get_refresh_relation_metadata(issue_ids)` raw-GQL helper is still needed for batched refresh. |
| **Workspace identity validation** â€” stable workspace id plus slug | Missing | **Still missing** â€” no workspace-parent identity surface on issue, team, user, or viewer paths | `get_issue_workspace_identity(issue_id)` raw-GQL helper is still required. |
| **Batch issue metadata for refresh** â€” by-id-set `updated_at` read | Missing | **Still missing** â€” no by-id-set batch metadata API | `get_refresh_issue_metadata(issue_ids)` raw-GQL helper is still required. |

### Remaining v1.1.0 Gaps Impacting Development

These gaps are confirmed absent in v1.1.0 and require raw-GraphQL fallback
in the gateway:

1. **Workspace identity** â€” no domain or GQL-service surface exposes workspace
   id plus slug. The gateway must issue a raw GraphQL query for
   `get_issue_workspace_identity(issue_id)`.

2. **Batch metadata reads** â€” the domain layer provides per-object reads only.
   The refresh flow requires metadata across the full tracked reachable set in
   one pass. The three batch helpers (`get_refresh_issue_metadata`,
   `get_refresh_comment_metadata`, `get_refresh_relation_metadata`) remain
   raw-GQL.

3. **File upload limitations** â€” `linear-client` v1.1.0 documents a known
   non-functional file-upload limitation (tracked as FW-22 in linear-client).
   This does not affect context-sync, which is read-only.

### v1.1.0 Breaking Changes Affecting context-sync

Current code impact is minimal â€” only
[src/context_sync/_cli.py:174â€“195](../../../src/context_sync/_cli.py) imports
`Linear`, and the real gateway is not yet implemented. The breaking changes
below affect the gateway implementation rather than existing code:

| Breaking change | Impact on gateway |
| --- | --- |
| `Issue.get_comments()` returns root-thread projection (not flat list) | Gateway must traverse `Comment.children` for full comment content |
| `Team.search_issues()` returns `IssueSearchPage` (not `list[Issue]`) | No impact â€” context-sync v1 does not use team-scoped search |
| `Issue.assign()` is keyword-only | No impact â€” gateway is read-only |
| `Issue.link()` removed; use `linear.issue_link().create()` | No impact â€” gateway is read-only |
| `IssuesService.link()` renamed to `IssuesService.create_link()` | No impact â€” gateway is read-only |
| `IssuesService.search(include_relations=...)` removed | Confirms the domain-layer `Issue.get_links()` path is correct |
| `User.is_app_actor()` removed; use `User.app` | Gateway should use `User.app` if it exposes actor type |
| `UsersService.look_up_id()` removed | No impact â€” gateway does not look up users by name |
| Constructor field validation rejects trusted-hydration-only fields | No impact â€” gateway reads, does not construct domain objects from raw data |
| Token file schema enforcement (`version` + `mode` required) | Operator-facing: legacy token files must be deleted and recreated |

### Agent-Awareness Improvements

`linear-client` v1.1.0 satisfies the agent-awareness requirements from
[docs/policies/common/python/agent-awareness.md](../../policies/common/python/agent-awareness.md):

- **`__init__.py` module docstring**: documents primary entry points, includes
  a runnable quick-start example, lists domain objects and semantic types
- **`__all__`**: explicit, exports only the intended public API
- **`py.typed`**: present
- **Type completeness**: 100% (`pyright --verifytypes` with `--ignoreexternal`)
- **Pyright mode**: `standard` with no suppressed rules
- **`examples/`**: three standalone contract-tested scripts
- **`docs/design/dependency-map.md`**: maps entry points, internal modules,
  workflows, and anti-patterns
- **Semantic type aliases**: 18 `NewType` aliases for identifiers, keys, URLs,
  and timestamps

The multi-repo workspace setup
([docs/workspace-setup.md](../../workspace-setup.md)) now includes
`linear-client` as a workspace folder, and the editable install gives pyright
and Pylance full type information across the dependency boundary.

  v1.1.0 (`Comment.updated_at`, `.resolved_at`, `.parent`, `.children`)

  conclusion. The M3-O1 core decision (canonical input accepted) is

  audit should state which comment-metadata reads can use the domain layer
  and which still require the raw-GQL batch path.
- The adapter-boundary definition must include a **type-mapping table**: for
  each value that crosses the gateway boundary, state whether it uses an
  upstream `linear_client.types` alias, a context-sync-only alias (with
  justification), or bare `str` (with justification). This makes the type
  surface an explicit boundary deliverable rather than an implementation-time
  afterthought.
- The audit must evaluate adopting `linear_client.types` as the authoritative
  source for the five context-sync types that are exact semantic duplicates
  of upstream aliases (`IssueId`, `IssueKey`, `CommentId`, `AttachmentId`,
  and `Timestamp`/`IsoTimestamp`) and recommend a concrete reconciliation
  strategy. The preferred approach is re-exporting from upstream (e.g.
  `from linear_client.types import IssueId as IssueId`) so that existing
  import sites continue to work while the underlying `NewType` identity is
  shared across the dependency boundary. For `Timestamp` vs. `IsoTimestamp`,
  the audit should evaluate whether to adopt the upstream name or keep an
  explicit alias. The three context-sync-only types (`WorkspaceId`,
  `WorkspaceSlug`, `WriterId`) remain local since they have no upstream
  counterpart.
- The audit must also evaluate whether context-sync should adopt additional
  upstream type aliases that are currently represented as bare `str`:
  `UserId`, `TeamId`, `TeamKey`, `StatusId`, `LabelId`, `IssueLinkId`,
  `IssueLinkType`, `StatusCategory`, and `AssetUrl`. Several of these appear
  in the gateway protocol or rendered output today as bare `str`.
- If the audit identifies required changes to existing done-ticket code, the
  `LinearGateway` protocol, or artifacts outside M5-1's scope, those must be
  represented as new follow-on tasks or scope amendments to existing Todo
  tickets rather than silently absorbed into M5-1 or left untracked.

from v1.0.0 will hit a predictable authentication failure after the upgrade.

- Add a note to M6-1 (supported user-facing docs and install/operator guide)
  that the operator guide must document recreating legacy token files for
  OAuth and client-credentials auth under `linear-client` v1.1.0. This
  applies only to those two auth modes; `api_key` mode does not use token
  files.
- Add a note to M5-2 (supported public runtime validation and smoke path)
  that the validation environment must use a freshly created v1.1.0 token
  file when exercising OAuth or client-credentials auth modes, so the smoke
  path confirms the migration is documented correctly.
