## Background

`linear-client` v1.1.0 has been released and is now available in the
multi-repo workspace. The release is a broad API-shape refinement that keeps
the two-layer architecture intact but makes the domain layer substantially
more complete and tightens several public contracts that were provisional in
v1.0.0. It also introduces intentional backward-incompatible changes
documented in its release notes.

The M1-D2 domain-coverage audit and the M5-1 ticket notes were written against
v1.0.0. This amendment updates those artifacts to reflect the v1.1.0 surface,
narrows the raw-GraphQL adapter boundary where v1.1.0 closes prior gaps, and
records the remaining gaps that still require raw-GraphQL helpers.

## v1.1.0 Gap Closure Analysis

The M1-D2 audit identified five categories of missing upstream capability in
v1.0.0. The table below records how v1.1.0 changes each:

| M1-D2 Gap | v1.0.0 Status | v1.1.0 Status | Impact on Adapter Boundary |
| --- | --- | --- | --- |
| **Traversal and rendered relations** — read-only issue-relation listing | Missing at both layers | **Closed** — `Issue.get_links()` returns `list[IssueLink]` with `link_type`, `updated_at`, `archived_at`; `IssueLink` is a full-CRUD domain object | Single-issue relation reads move to the domain layer. The `get_ticket_relations(issue_ids)` raw-GQL helper is only needed for batched multi-issue reads if the gateway batches relation fetches across the tracked set. |
| **Comment metadata for `comments_signature`** — `updatedAt`, parent/root topology, thread `resolved`, deletion signal | Missing at the domain layer | **Partially closed** — `Comment.updated_at`, `.resolved_at`, `.parent`, `.children`, `.is_resolved()`, `.is_placeholder()` are now domain-layer; however, no batch-by-issue-set comment-metadata endpoint exists | Per-issue comment metadata is domain-layer. The `get_refresh_comment_metadata(issue_ids)` raw-GQL helper is still needed for the batched refresh-cursor pass across the tracked reachable set. |
| **Relation metadata for `relations_signature`** — per-issue relation reads | Missing at both layers | **Partially closed** — `Issue.get_links()` provides per-issue relation metadata including `updated_at` and `archived_at`; no batch-by-issue-set relation-metadata endpoint exists | Same pattern as comments: per-issue reads are domain-layer, but the `get_refresh_relation_metadata(issue_ids)` raw-GQL helper is still needed for batched refresh. |
| **Workspace identity validation** — stable workspace id plus slug | Missing | **Still missing** — no workspace-parent identity surface on issue, team, user, or viewer paths | `get_issue_workspace_identity(issue_id)` raw-GQL helper is still required. |
| **Batch issue metadata for refresh** — by-id-set `updated_at` read | Missing | **Still missing** — no by-id-set batch metadata API | `get_refresh_issue_metadata(issue_ids)` raw-GQL helper is still required. |

### Summary of Boundary Change

The raw-GraphQL adapter boundary defined by M1-D2 §2 shrinks. After this
amendment, raw `linear.gql.*` usage inside the gateway adapter is approved only
for:

1. `get_issue_workspace_identity(issue_id)` — unchanged
2. `get_refresh_issue_metadata(issue_ids)` — unchanged
3. `get_refresh_comment_metadata(issue_ids)` — unchanged (batch reads only;
   per-issue comment metadata should use the domain layer)
4. `get_refresh_relation_metadata(issue_ids)` — unchanged (batch reads only;
   per-issue relation reads should use the domain layer)
5. ~~`get_ticket_relations(issue_ids)`~~ — **narrowed**: per-ticket relation
   reads during sync/traversal must use `Issue.get_links()` at the domain
   layer. A raw-GQL batch helper is only approved if M5-1 implementation
   confirms that issuing one `get_links()` call per tracked ticket is
   materially more expensive than a single batched query, and only for
   that batch path.

### Additional v1.1.0 Capabilities Available to M5-1

These v1.1.0 features are not required by the current v1 design but are
available for M5-1 to use where they simplify the gateway implementation or
enrich rendered output:

| Capability | Domain surface | Potential use |
| --- | --- | --- |
| **Status workflow categories** | `Status.category` (`"triage"`, `"backlog"`, `"unstarted"`, `"started"`, `"completed"`, `"canceled"`) | Enrich rendered ticket metadata with workflow state category |
| **Issue team relationship** | `Issue.team`, `Issue.get_team()` | Render team context on ticket files |
| **Delegation** | `Issue.delegate`, `Issue.get_delegate()`, `Issue.assign(delegate=...)` | Render delegate alongside assignee |
| **App-actor identification** | `User.app` (boolean flag, replaces removed `is_app_actor()` heuristic) | Distinguish app-actor vs human in rendered metadata |
| **Issue history** | `Issue.get_history()`, `get_last_status()`, `get_last_assignee()` | Not needed for v1; connects to [FW-5](../../future-work.md#fw-5-ticket-history-and-sectioned-ticket-artifacts) |
| **Previous issue keys** | `Issue.previous_keys` | Not needed for v1; connects to [FW-4](../../future-work.md#fw-4-historical-ticket-alias-import) |
| **Paginated search** | `Team.search_issues() -> IssueSearchPage` with cursor pagination | No current v1 use; available for future features |
| **Semantic type aliases** | `IssueId`, `IssueKey`, `UserId`, `TeamId`, etc. (18 `NewType` aliases) | Adopt in context-sync's own `_types.py` for cross-boundary type safety |
| **100% type completeness** | `py.typed` + pyright `standard` mode with no suppressions | Remove `# type: ignore[import-untyped]` from `_cli.py`; full pyright coverage of gateway code |
| **`LinearNotFoundError`** | New exception subclass for entity-not-found | Use for explicit not-found handling in the gateway |

### Remaining v1.1.0 Gaps Impacting Development

These gaps are confirmed absent in v1.1.0 and require raw-GraphQL fallback
in the M5-1 gateway:

1. **Workspace identity** — no domain or GQL-service surface exposes workspace
   id plus slug. The gateway must issue a raw GraphQL query for
   `get_issue_workspace_identity(issue_id)`.

2. **Batch metadata reads** — the domain layer provides per-object reads only.
   The refresh flow requires metadata across the full tracked reachable set in
   one pass. The three batch helpers (`get_refresh_issue_metadata`,
   `get_refresh_comment_metadata`, `get_refresh_relation_metadata`) remain
   raw-GQL.

3. **File upload limitations** — `linear-client` v1.1.0 documents a known
   non-functional file-upload limitation (tracked as FW-22 in linear-client).
   This does not affect context-sync, which is read-only.

### v1.1.0 Breaking Changes Affecting context-sync

Current code impact is minimal — only
[src/context_sync/_cli.py:174–195](../../../src/context_sync/_cli.py) imports
`Linear`, and the real gateway is not yet implemented. The breaking changes
below affect the M5-1 implementation rather than existing code:

| Breaking change | Impact on M5-1 |
| --- | --- |
| `Issue.get_comments()` returns root-thread projection (not flat list) | Gateway must traverse `Comment.children` for full comment content |
| `Team.search_issues()` returns `IssueSearchPage` (not `list[Issue]`) | No impact — context-sync v1 does not use team-scoped search |
| `Issue.assign()` is keyword-only | No impact — gateway is read-only |
| `Issue.link()` removed; use `linear.issue_link().create()` | No impact — gateway is read-only |
| `IssuesService.link()` renamed to `IssuesService.create_link()` | No impact — gateway is read-only |
| `IssuesService.search(include_relations=...)` removed | Confirms the domain-layer `Issue.get_links()` path is correct |
| `User.is_app_actor()` removed; use `User.app` | Gateway should use `User.app` if it exposes actor type |
| `UsersService.look_up_id()` removed | No impact — gateway does not look up users by name |
| Constructor field validation rejects trusted-hydration-only fields | No impact — gateway reads, does not construct domain objects from raw data |
| Token file schema enforcement (`version` + `mode` required) | Operator-facing: legacy token files must be deleted and recreated |

### Agent-Awareness Improvements

`linear-client` v1.1.0 satisfies the agent-awareness requirements from
[docs/policies/common/python/agent-awareness.md](../../policies/common/python/agent-awareness.md):

- **`__init__.py` module docstring**: documents primary entry points, includes
  a runnable quick-start example, lists domain objects and semantic types
- **`__all__`**: explicit, exports only the intended public API
- **`py.typed`**: present
- **Type completeness**: 100% (`pyright --verifytypes` with `--ignoreexternal`)
- **Pyright mode**: `standard` with no suppressed rules
- **`examples/`**: three standalone contract-tested scripts
- **`docs/design/dependency-map.md`**: maps entry points, internal modules,
  workflows, and anti-patterns
- **Semantic type aliases**: 18 `NewType` aliases for identifiers, keys, URLs,
  and timestamps

The multi-repo workspace setup
([docs/workspace-setup.md](../../workspace-setup.md)) now includes
`linear-client` as a workspace folder, and the editable install gives pyright
and Pylance full type information across the dependency boundary.

## Why This Amendment Fits the Planning Model

This is a material amendment under
[docs/policies/common/planning-model.md §5](../../policies/common/planning-model.md#5-material-amendments)
because it changes the dependency version that design and implementation
artifacts reference, narrows the approved adapter boundary (which changes the
implementation shape of M5-1), and updates an existing design artifact.

This amendment follows the done-ticket amendment rule: M1-D2 is cited as a
completed design ticket whose artifact is updated with a factual correction
(the upstream package version changed and its coverage expanded), not a
retroactive expansion of the M1-D2 deliverable. The boundary direction
(domain-layer-first with a narrow raw-GQL escape hatch) is unchanged; only
the coverage state and the list of approved raw-GQL helpers are updated.

No new tickets are added, no milestones are added or removed, and no
prerequisites are reordered.
