#### 2.2 Detailed Ticket Notes

##### M4.1-1 - CLI and library simplification

**CLI changes (from [M4-R1](../../execution/M4-R1.md) definitive proposal):**

- Remove the `add` subcommand from the CLI parser. An operator invoking
  `context-sync add` receives the standard unknown-subcommand error; no
  special redirect or educational message is needed.
- Rename the `remove-root` subcommand to `remove`. The old `remove-root`
  token is likewise handled by the standard unknown-subcommand error path.
- Use `TICKET` as the single positional-argument placeholder and help
  description across `sync` and `remove`, described as "Issue key or Linear
  URL of the ticket to track or remove."
- Update the `sync` help string from the current implementation-oriented
  "Full-snapshot sync from a root ticket" to intent-oriented text such as
  "Track a ticket and fetch its full issue hierarchy."
- Update the `refresh` help string from "Incremental refresh of all tracked
  roots" to intent-oriented text such as "Re-fetch the latest data for all
  tracked tickets." The two strings must use clearly distinct language so
  that the semantic difference is apparent to callers: `sync` performs a
  full replacement of all nodes regardless of existing data or timestamps;
  `refresh` is a lightweight pass that only updates nodes identified as new
  or modified since the last run.
- `sync` remains the only root-expanding public command. The
  full-vs-incremental execution strategy remains entirely internal per the
  [M4-R1 §3a](../../execution/M4-R1.md) hidden-strategy recommendation.

**Library changes (from [M4-R2](../../execution/M4-R2.md) §3, §4):**

- Make `ContextSync.add()` internal: rename to `_add()` and remove from
  `__all__`. The method implementation may be retained as an internal helper
  behind `sync()` if that is the cleanest factoring.
- Rename `ContextSync.remove_root()` to `ContextSync.remove()`.
- When `sync()` is called without explicit `max_tickets_per_root` or
  `dimensions` overrides, it must preserve the manifest's existing traversal
  configuration for the supplied root rather than overwriting it with
  defaults. When overrides are explicitly supplied, persist them as today.
  This adopts `add()`'s preservation semantics as the no-override default,
  per [M4-R1 §definitive-proposal item 4](../../execution/M4-R1.md) and
  [M4-R1-R4](../../execution/M4-R1-review.md).
- Update the `SyncResult` docstring to remove the reference to `add()` if it
  is no longer public.
- The documentation-only docstring improvements identified by M4-R2 (§5
  summary-line rewording for `sync()` and `refresh()`) should land alongside
  the method boundary changes since the surface is changing in the same
  ticket.

**Documentation changes:**

- Update [README.md](../../../README.md) to reflect the four-command surface.
  Remove the interim "When to use each command" section (added for
  [M4-R1-R1](../../execution/M4-R1-review.md)) and replace with documentation
  of the simplified model.
- Update all user-facing command tables, exit-code tables, and examples.
- Ensure the supported docs surface is consistent before M5-2 or M6-1
  proceeds.

#### 2.3 Exit Criteria

1. The public CLI surface is `sync`, `refresh`, `remove`, and `diff`. The old
   `add` and `remove-root` tokens are rejected by the standard
   unknown-subcommand error path.
2. The public library surface exposes `ContextSync.sync()`,
   `ContextSync.refresh()`, `ContextSync.remove()`, and `ContextSync.diff()`.
   `add()` is no longer in `__all__`.
3. `sync()` preserves existing manifest traversal configuration when no
   overrides are supplied. Explicit overrides are persisted as before.
4. README and user-facing docs reflect the four-command model with no
   references to the old `add` or `remove-root` surface.

## Why This Amendment Fits the Planning Model

This is a material amendment under
[docs/policies/common/planning-model.md §5](../../policies/common/planning-model.md#5-material-amendments)
because it:

- adds a new named milestone and plan item (`M4.1-1`)
- pulls deferred backlog work ([FW-10](../../future-work.md#fw-10-cli-simplification-amendment))
  into the active plan
- changes delivery shape by inserting a new prerequisite before M5

This amendment follows the done-ticket amendment rule: M4-R1, M4-R2, and M4-3
are cited as satisfied prerequisites whose accepted repository artifacts
already support the new dependency. No completed ticket is being widened.
