#### 2.2 Detailed Ticket Notes

##### M4.1-1 - CLI and library simplification

**Use-case coverage:**

The four-command surface after this ticket must cover all six operator use
cases. The table below is the authoritative mapping:

| # | Use case | CLI command | Notes |
| --- | --- | --- | --- |
| 1 | Add the first tracked root | `sync TICKET` | Bootstraps the snapshot; TICKET becomes the initial root |
| 2 | Add a new root to an existing snapshot | `sync TICKET` | Same command; idempotent if TICKET is already tracked |
| 3 | Lightweight incremental refresh | `refresh` | Re-fetches only nodes identified as new or modified since the last run |
| 4 | Full rebuild (nuke and rebuild) | `sync` | Rebuilds all tracked roots unconditionally; no root-membership change |
| 5 | Remove a root and prune orphaned nodes | `remove TICKET` | Stops tracking TICKET and removes nodes not reachable from any remaining root |
| 6 | Diff without writing to disk | `diff` | Read-only comparison between local snapshot and Linear |

Note: both `sync TICKET` and standalone `sync` currently rebuild the full root
set regardless of graph connectivity. Selective partition-aware rebuilding is
deferred to
[FW-11](../../future-work.md#fw-11-partition-aware-sync-rebuild).

**CLI changes (from [M4-R1](../../execution/M4-R1.md) definitive proposal):**

- Remove the `add` subcommand from the CLI parser. An operator invoking
  `context-sync add` receives the standard unknown-subcommand error; no
  special redirect or educational message is needed.
- Rename the `remove-root` subcommand to `remove`. The old `remove-root`
  token is likewise handled by the standard unknown-subcommand error path.
- `sync`'s positional argument `TICKET` is optional. When supplied, `sync`
  adds or reaffirms TICKET as a tracked root then fully rebuilds all tracked
  roots. When omitted, `sync` performs the same full rebuild without any
  root-membership change (use case 4). `remove`'s TICKET argument remains
  required.
- Use `TICKET` as the positional-argument placeholder for both `sync` and
  `remove`. The `sync` argument description should convey its optional nature:
  "Issue key or Linear URL of the ticket to add or reaffirm as a root
  (optional; omit to rebuild all roots without changing root membership)."
  For `remove`: "Issue key or Linear URL of the tracked root to remove."
- Update the `sync` help string from the current implementation-oriented
  "Full-snapshot sync from a root ticket" to intent-oriented text that covers
  both modes, for example: "Fully rebuild the snapshot from all tracked
  roots. With TICKET, add or reaffirm that root first."
- Update the `refresh` help string from "Incremental refresh of all tracked
  roots" to intent-oriented text such as "Re-fetch the latest data for all
  tracked tickets." The two strings must use clearly distinct language so
  that the semantic difference is apparent to callers: `sync` unconditionally
  rewrites all nodes regardless of existing data or timestamps; `refresh` is a
  lightweight pass that only updates nodes identified as new or modified since
  the last run.
- `sync` remains the only root-expanding public command. The
  full-vs-incremental execution strategy remains entirely internal per the
  [M4-R1 §3a](../../execution/M4-R1.md) hidden-strategy recommendation.

**Library changes (from [M4-R2](../../execution/M4-R2.md) §3, §4):**

- Make `ContextSync.add()` internal: rename to `_add()` and remove from
  `__all__`. The method implementation may be retained as an internal helper
  behind `sync()` if that is the cleanest factoring.
- Rename `ContextSync.remove_root()` to `ContextSync.remove()`.
- Make `sync()`'s `key` parameter optional (`key: str | None = None`). When
  `key` is `None`, `sync()` performs a full rebuild of all currently tracked
  roots with no root-membership change — the library-level equivalent of
  standalone CLI `sync`. When `key` is supplied, behavior is unchanged: add
  or reaffirm the root, then full-rebuild all roots. There is no expectation
  of a 1:1 CLI-to-library mapping; the CLI dispatcher may call either form
  as appropriate.
- When `sync()` is called without explicit `max_tickets_per_root` or
  `dimensions` overrides, it must preserve the manifest's existing traversal
  configuration for the supplied root rather than overwriting it with
  defaults. When overrides are explicitly supplied, persist them as today.
  This adopts `add()`'s preservation semantics as the no-override default,
  per [M4-R1 §definitive-proposal item 4](../../execution/M4-R1.md) and
  [M4-R1-R4](../../execution/M4-R1-review.md).
- Update the `SyncResult` docstring to remove the reference to `add()` if it
  is no longer public.
- The documentation-only docstring improvements identified by M4-R2 (§5
  summary-line rewording for `sync()` and `refresh()`) should land alongside
  the method boundary changes since the surface is changing in the same
  ticket.

**Documentation changes:**

- Update [README.md](../../../README.md) to reflect the four-command surface.
  Remove the interim "When to use each command" section (added for
  [M4-R1-R1](../../execution/M4-R1-review.md)) and replace with documentation
  of the simplified model.
- Update all user-facing command tables, exit-code tables, and examples.
- Ensure the supported docs surface is consistent before M5-2 or M6-1
  proceeds.

#### 2.3 Exit Criteria

1. The public CLI surface is `sync [TICKET]`, `refresh`, `remove TICKET`, and
   `diff`. The old `add` and `remove-root` tokens are rejected by the standard
   unknown-subcommand error path.
2. `sync` with no TICKET argument performs a full rebuild of all currently
   tracked roots without changing root membership (use case 4).
3. The public library surface exposes `ContextSync.sync()` (with optional
   `key`), `ContextSync.refresh()`, `ContextSync.remove()`, and
   `ContextSync.diff()`. `add()` is no longer in `__all__`.
4. `sync()` preserves existing manifest traversal configuration when no
   overrides are supplied. Explicit overrides are persisted as before.
5. README and user-facing docs reflect the four-command surface, document all
   six use cases, and contain no references to the old `add` or `remove-root`
   surface.
