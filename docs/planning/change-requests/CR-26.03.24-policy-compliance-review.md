## Reviewer Notes

### Gap Analysis Methodology

The CR consulted the right policy sources and correctly identified seven
tooling/artifact gaps. Where the analysis fell short was in not independently
verifying the codebase against every requirement within those sources. The gap
table captures what is absent from the repository (missing files, missing
config, missing tools) but does not audit what the existing code delivers
against the policy requirements that apply to that code. The two high-severity
findings (CR-26.03.24-PC-R1 and CR-26.03.24-PC-R2) are both code-level
compliance gaps that a repository-state-only scan would not surface.

### Verified Gaps (G-1 through G-6)

All six non-README gaps are confirmed against current repository state:

- **G-1** (`scripts/validate.sh`): `scripts/` contains only `.linear_env.sh`
  and its sample — no quality gate entry point.
- **G-2** (Pyright): zero occurrences of `pyright` in `pyproject.toml`.
- **G-3** (`py.typed`): `src/context_sync/py.typed` does not exist.
- **G-4** (`coverage.py` / `pytest-cov`): zero occurrences of `coverage` or
  `pytest-cov` in `pyproject.toml`.
- **G-5** (`examples/`): directory does not exist at repository root.
- **G-6** (`docs/design/dependency-map.md`): file does not exist.

### Plan-Covered and Advisory Classifications

The five plan-covered gaps (release scripts, CHANGELOG, MkDocs, release
checklist, `pyright --verifytypes`) are correctly attributed to M6-1, M6-2,
and M6-R1. The five advisory items (data-modeling, code-review checklist,
design-review checklist, implementation-plan template, release-checklist
template) are correctly classified — none creates a retroactive compliance gap.

Independent audit confirmed that the data-modeling decision rule is already
followed: Pydantic `BaseModel` is used for serialization-boundary models
(`LockRecord`, `Manifest`, `ManifestRootEntry`, `ManifestTicketEntry`,
`ManifestSnapshot`) and frozen `dataclass` is used for internal state
containers (`IssueData`, `CommentData`, `SyncResult`, `TraversedTicket`,
etc.). No plain `dict` is used as an ad-hoc data model. The exception
hierarchy in
[src/context_sync/\_errors.py](../../../src/context_sync/_errors.py) is
well-structured, exported via `__all__`, and has a proper module docstring.
Basic type annotations (parameters, return types, `from __future__ import
annotations`) are complete across all modules.

### Scope Fit

The proposed M4.2 milestone is correctly scoped. Placing it between M4.1
(complete) and M5 (not started) respects the done-ticket amendment rule and
ensures the quality-gate baseline is established before later milestones build
on it. The CR's rationale for rejecting the three alternatives (ad hoc work,
fold into M5, reopen M4.1) is sound.

### Candidate Decisions

All four candidate decisions are well-reasoned:

- Rejecting ad hoc work correctly recognizes the gaps are material enough for
  tracked execution artifacts.
- Rejecting M5 absorption correctly separates toolchain/compliance work from
  gateway/runtime work.
- Rejecting M4.1 reopening correctly respects the done-ticket amendment rule.
- Keeping a new M4.2 provides a clean milestone boundary with minimal
  disruption.

### Done-Ticket Amendment Rule

The CR complies. M4.1-2 is cited as a satisfied prerequisite for M4.2-1. The
CR does not broaden M4.1-2's deliverable, dependency obligations, or exit
criteria. The dependency is explicit and the relationship is accurately
described as predecessor, not retroactive expansion.

### Sequencing and Dependencies

The dependency chain is sound for the proposed tickets:

- M4.2-1 depends on M4.1-2 (Done) — can start immediately.
- M4.2-2 depends on M4.2-1 — correct because coverage tooling should be
  validated through `validate.sh`.
- M5-1 gaining M4.2-1 as a dependency is the right call.

The dependency completeness question (CR-26.03.24-PC-R6) is whether M4.2-2
should also feed into M5-2.

### Exit Criteria

The seven exit criteria are clear and verifiable. Each maps to one or more
gaps. If the CR is amended to add G-8 (semantic types) and G-9 (docstring
dimensions), the exit criteria should be extended to cover:

- `_types.py` exists and defines `NewType` aliases for the identified domain
  concepts
- Pyright validates the `NewType` boundaries with zero errors
- All four `ContextSync` public methods document the required dimensions

## Residual Risks

- **Pyright error volume.** G-2 resolution (Pyright zero errors) is the
  highest-risk deliverable in M4.2-1. If the codebase has many latent type
  inconsistencies that Ruff's `ANN` rules did not catch, resolving them all
  in one ticket could be time-consuming. The ticket notes correctly suggest
  starting with `typeCheckingMode = "standard"` and tightening only if
  achievable — this is the right approach, but the ticket owner should assess
  the Pyright error count early and flag if the volume is unexpectedly high.
- **Semantic type adoption scope.** If CR-26.03.24-PC-R1 is accepted and
  `NewType` aliases are introduced in M4.2-1, the type migration touches
  every module that handles issue IDs, issue keys, workspace IDs, and comment
  IDs. This is a wide mechanical change. The risk is that the migration
  surfaces Pyright errors that are actually logic bugs (passing an `IssueKey`
  where an `IssueId` is expected), which would be good to discover but could
  expand the ticket's scope. The ticket owner should plan for this and scope
  the initial `NewType` set to the minimum that closes the policy gap rather
  than attempting a maximal migration.
- **Dependency map freshness.** The dependency map created in M4.2-2 will
  describe the pre-M5 codebase. When M5-1 introduces the real gateway, the
  dependency map's entry-point and workflow-to-code sections will need
  updating. The agent-awareness policy requires the map to be "updated as
  part of any change that adds a new public workflow, renames a key module,
  or changes the primary entry point"
  ([python/agent-awareness.md:196–197](../../policies/common/python/agent-awareness.md)),
  so this is already covered by policy — but it is worth noting that the
  M4.2-2 artifact is not a write-once deliverable.
