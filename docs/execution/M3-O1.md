# M3-O1 - `comments_signature` input settlement

> **Status**: Done

## Initial Questions

None.

## Operational-State Capture

### Background

[M1-D3](M1-D3.md) defined the v1 composite refresh cursor with a
`comments_signature` component computed from a canonical input that includes
per-comment `updated_at`, per-thread `resolved` state, and parent/root topology.

[M1-D2](M1-D2.md) audited the `linear-client` domain layer and confirmed it
does **not** expose the fields required by that canonical input: comment
`updatedAt`, parent/root topology, thread `resolved`, or a deletion signal.
However, M1-D2 did not probe whether the raw Linear GraphQL API — the boundary
already designated for refresh-metadata adapters — exposes those fields.

Review finding [M1-D2-R1](M1-D2-review.md) (Medium, Fix now) flagged the
unresolved fallback. The implementation plan was amended to carry this forward
as a pre-M3-1 gate
([M3-O1](../implementation-plan.md#m3-o1---comments-signature-input-settlement)),
requiring either probe-backed confirmation of the existing canonical input or
an accepted design amendment naming a replacement.

### Resolution path taken

**Path (a): probe the raw Linear GraphQL API.** The probe confirmed that all
required fields are available and that `updatedAt` advances on the mutation
events that matter for `comments_signature` correctness.

## Probe Evidence

### Schema introspection

Introspection of the `Comment` type in the Linear GraphQL API
(`2026-03-21T15:06Z`) confirmed the following fields relevant to
`comments_signature`:

| Field | GraphQL type | Nullability | Purpose |
|---|---|---|---|
| `id` | `ID` | non-null | Stable comment identity |
| `updatedAt` | `DateTime` | non-null | Change signal for edit, resolve, reply |
| `editedAt` | `DateTime` | nullable | Body-edit-specific timestamp |
| `createdAt` | `DateTime` | non-null | Creation timestamp |
| `parentId` | `String` | nullable | Parent comment ID (thread topology) |
| `parent` | `Comment` | nullable | Parent comment object |
| `children` | `CommentConnection` | non-null | Child comments |
| `resolvedAt` | `DateTime` | nullable | Thread resolved signal (null = open) |
| `resolvingComment` | `Comment` | nullable | Comment that triggered resolution |
| `archivedAt` | `DateTime` | nullable | Soft-delete / tombstone signal |
| `body` | `String` | non-null | Comment body text |

Thread resolution is controlled by dedicated mutations (`commentResolve`,
`commentUnresolve`), not by a writable field on `commentUpdate`.

### Behavioral probes

All probes ran against the `ARK` test-harness workspace using disposable issues
that were archived after each probe. Probes used 3-second waits between
mutations to rule out stale-read artifacts.

#### Probe 1: `updatedAt` on comment body edit (ARK-16)

| Step | `updatedAt` | `editedAt` |
|---|---|---|
| Comment created | `2026-03-21T15:06:12.741Z` | `None` |
| Body edited | `2026-03-21T15:06:15.801Z` | `2026-03-21T15:06:15.800Z` |

**Result:** `updatedAt` advanced by ~3.06 s. `editedAt` populated on first body
edit.

#### Probe 2: `updatedAt` on thread resolve and unresolve (ARK-17)

| Step | `updatedAt` | `resolvedAt` |
|---|---|---|
| Comment created | `2026-03-21T15:07:10.633Z` | `None` |
| Thread resolved | `2026-03-21T15:07:13.824Z` | `2026-03-21T15:07:13.824Z` |
| Thread unresolved | `2026-03-21T15:07:16.851Z` | `None` |

**Result:** `updatedAt` advanced on both resolve (~3.19 s) and unresolve
(~3.03 s). `resolvedAt` is set on resolve and cleared on unresolve.

#### Probe 3: parent `updatedAt` on child reply creation (ARK-17)

| Step | Parent `updatedAt` |
|---|---|
| Before reply | `2026-03-21T15:07:16.851Z` |
| After reply | `2026-03-21T15:07:19.925Z` |

**Result:** Parent comment `updatedAt` advanced (~3.07 s) when a child reply
was created. Reply `parentId` correctly references the parent comment ID.

### Summary of behavioral findings

| Mutation event | `updatedAt` advances? | Notes |
|---|---|---|
| Comment body edit | **Yes** | `editedAt` also populated |
| Thread resolve (`commentResolve`) | **Yes** | `resolvedAt` set to same timestamp |
| Thread unresolve (`commentUnresolve`) | **Yes** | `resolvedAt` cleared to null |
| Child reply created | **Yes** (parent) | Parent `updatedAt` advances; child has own `updatedAt` |

## Accepted Decision

The [M1-D3](M1-D3.md) `comments_signature` canonical input is **accepted as
originally specified**. No design amendment is required.

The raw Linear GraphQL API exposes all fields needed by the canonical input
record defined in the [snapshot consistency
contract](../adr.md#61-snapshot-consistency-contract):

- **`comment|<comment_id>|root=<root_comment_id>|parent=<parent_or_none>|updated_at=<timestamp>|deleted=<bool_or_unknown>`**:
  `id`, `parentId`, `updatedAt`, and `archivedAt` (as deletion proxy) are all
  available. Root comment identity is derivable from topology (`parentId` is
  null for root comments).

- **`thread|<root_comment_id>|resolved=<bool>`**: `resolvedAt` is available on
  root comments. The boolean maps to `resolvedAt is not None`.

The adapter boundary defined by [M1-D2](M1-D2.md) — local raw `linear.gql.*`
calls for batched refresh-metadata queries — is the correct integration point.
These fields are **not** available through the `linear-client` domain layer,
confirming M1-D2's audit and the need for the raw-adapter path.

### Deletion signal mapping

The canonical input specifies `deleted=<bool_or_unknown>`. Linear comments
support `archivedAt` (soft archive) and `commentDelete` (hard delete). For
hard-deleted comments, the comment will simply be absent from the query
response. For soft-archived comments, `archivedAt` is non-null.

The `deleted` canonical input maps to:

- `true` if `archivedAt` is non-null (archived/soft-deleted)
- `unknown` if the comment was previously tracked but is absent from the
  current query response (hard-deleted or access-revoked)

Comments that were never tracked and are simply absent are not included in the
canonical input at all (they are newly discovered, not deleted).

## Verification

- Schema introspection confirms field availability (non-null `updatedAt`,
  nullable `parentId`, `resolvedAt`, `archivedAt`).
- Three independent behavioral probes confirm `updatedAt` advances on all
  mutation events relevant to `comments_signature` correctness.
- Probe issues archived after each run; no test-harness queue clutter remains.

**Unblocks:**
[M3-1](../implementation-plan.md#m3-1---incremental-refresh-and-quarantined-root-recovery)
(Incremental `refresh` and quarantined-root recovery).

## Completion Summary

The `comments_signature` input gate is resolved. The Linear GraphQL API
provides all fields required by the M1-D3 canonical input specification, and
behavioral probes confirm that comment `updatedAt` advances on body edits,
thread resolution changes, and child reply creation. The existing canonical
input is accepted without amendment. M3-1 may proceed.
