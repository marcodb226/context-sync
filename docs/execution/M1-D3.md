  to scope the amendment.
- Confirmed that the pre-amendment ADR and top-level design still described the
  invalidated single-cursor refresh design. Chose to rewrite both the
  governing design section and the ADR refresh-strategy prose so later readers
  do not have to reconstruct the accepted contract only from the
  [OQ-1](../adr.md#oq-1-refresh-freshness-validation-against-live-linear-behavior)
  outcome note.
- Chose the v1 composite refresh cursor as three explicit components:
  `issue_updated_at`, `comments_signature`, and `relations_signature`.
  Recorded that local ticket files persist those values under a
  machine-readable `refresh_cursor` mapping and that `refresh` compares the
  local and remote cursors by exact equality rather than by a monotonic
  `updated_at > last_synced_at` test.
- Defined the comment freshness contract conservatively as a deterministic
  digest over visible comment/thread metadata needed by the rendered comments
  section: stable comment IDs, parent/root relationships, comment `updated_at`,
  and per-thread `resolved` state. This satisfies the plan requirement to
  detect comment creation and comment edits and also keeps thread-state changes
  inside the same cursor component.
- Explicitly chose not to run additional live relation probes in this ticket.
  The plan notes treat relation probing as recommended input rather than a
  mandatory subtask, so the amendment instead documents a conservative
  first-release rule: relation freshness is an independent cursor component and
  must not rely on an unverified assumption that relation changes advance the
  parent issue `updated_at`.
- Defined relation freshness as a deterministic digest over the visible
  persisted relation tuples that affect rendered frontmatter and graph
  reachability. Recorded that the canonical input must include dimension,
  relation type, and target identity, using stable target UUIDs for canonical
  ordering when available while still including the current human-readable
  target key because that value is rendered locally.
- Narrowed attachment freshness out of the v1 selective-refresh correctness
  contract while keeping attachment metadata persisted in ticket files.
  Recorded that attachment-only drift is not guaranteed to be detected during
  selective refresh and remains deferred to
  [FW-2](../future-work.md#fw-2-attachment-content-handling).
- Recorded the exact remote-data requirements that
  [M1-D2](../implementation-plan.md#m1-d2---linear-domain-coverage-audit-and-adapter-boundary)
  must audit: issue identity plus issue `updated_at`, comment/thread metadata
  sufficient to build `comments_signature`, and relation metadata sufficient to
  build `relations_signature`, all behind one narrow batched metadata adapter
  path instead of one full ticket fetch per tracked issue.
- Updated
  [docs/design/0-top-level-design.md](../design/0-top-level-design.md#62-refresh-flow)
  to replace the invalidated single-cursor refresh flow with the accepted
  composite-cursor design and to record the explicit `M1-D2` audit inputs.
- Updated
  [docs/adr.md](../adr.md#52-refresh-incremental-whole-snapshot-update)
  so the ADR refresh strategy now matches the accepted composite-cursor
  contract instead of leaving the stale issue-level-only design in place.
- Updated
  [docs/design/refresh-freshness-validation.md](../design/refresh-freshness-validation.md)
  to add a forward pointer from the historical spike outcome to the accepted
  [M1-D3](../implementation-plan.md#m1-d3---refresh-composite-freshness-contract-amendment)

  `issue_updated_at`, `comments_signature`, and `relations_signature`;
  comparison is exact equality against the local `refresh_cursor`;
  `last_synced_at` remains informational rather than the staleness test; and
  attachment-only metadata drift is explicitly outside the v1 selective-refresh
  correctness contract.
- ADR update scope decision:
  this ticket **did** rewrite
  [docs/adr.md](../adr.md#52-refresh-incremental-whole-snapshot-update)
  because leaving the stale single-cursor prose in place would make the ADR
  and top-level design disagree about the active refresh contract.
- Conservative-design note:
  no additional live relation probes were run in this ticket. The design makes
  that explicit and resolves the uncertainty by requiring independent relation
  freshness rather than by assuming favorable upstream behavior.
- Validation notes:
  this was a docs-only design ticket, so no repository-wide lint, format, or
  test commands were run. Validation consisted of manual cross-document
  consistency checks across the governing plan, ADR, top-level design,
