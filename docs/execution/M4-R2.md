### Review conclusions

#### 1. The `_id` naming problem is wider than M4-3 covers

[M4-3](../implementation-plan.md#m4-3---rename-root-ticket-id-to-key) targets
only `sync(root_ticket_id=...)`. The same `_id` misnomer appears in two
additional public types:

| Location | Current field | Actual value | Problem |
| --- | --- | --- | --- |
| [src/context_sync/_sync.py:292](../../src/context_sync/_sync.py#L292) | `root_ticket_id` | Issue key or Linear URL | M4-3 already covers this |
| [src/context_sync/_models.py:33](../../src/context_sync/_models.py#L33) | `SyncError.ticket_id` | Issue key (docstring says "Current issue key for human-readable reporting") | Not covered by M4-3 |
| [src/context_sync/_models.py:78](../../src/context_sync/_models.py#L78) | `DiffEntry.ticket_id` | Issue key (docstring says "Current issue key for reporting") | Not covered by M4-3 |

A caller reading `error.ticket_id` reasonably expects an internal UUID, not
`"ACP-123"`. The rename should cover all three sites.

Recommended names:
- `sync(root_ticket_id=...)` → `sync(key=...)` (as M4-3 already proposes)
- `SyncError.ticket_id` → `SyncError.ticket_key`
- `DiffEntry.ticket_id` → `DiffEntry.ticket_key`

#### 2. Parameter naming is inconsistent across methods

The three root-accepting methods use two different parameter names for
the same concept:

| Method | Parameter | Accepts |
| --- | --- | --- |
| `sync()` | `root_ticket_id` | Issue key or Linear URL |
| `add()` | `ticket_ref` | Issue key or Linear URL |
| `remove_root()` | `ticket_ref` | Issue key or Linear URL |

`ticket_ref` is more accurate than `root_ticket_id`, but neither matches the
CLI's recommended `TICKET` placeholder from
[M4-R1](M4-R1.md#definitive-proposal). The library should settle on one
parameter name for this concept.

Recommended name: `key`. This is what M4-3 already proposes for `sync()`, and
it should apply uniformly to `add()` and `remove_root()` as well. The
parameter name `key` is short, accurate (the value is a human-facing key or a
URL that resolves to one), and aligns with the design document's terminology
("issue key").

#### 3. `add()` should become internal when the CLI simplification lands

[M4-R1](M4-R1.md) recommends removing the public `add` CLI action and folding
its user intent into `sync`. The library question is whether
`ContextSync.add()` should remain in the public API.

Assessment: `add()` should become an internal helper (`_add()`) or be removed
once `sync()` subsumes its role. The reasons:

1. If the CLI no longer exposes `add` as a distinct command, having a separate
   public library method with its own name and parameter signature creates a
   CLI/library divergence that confuses callers who use both.
2. The behavioral difference between `sync()` and `add()` is an implementation
   detail (full-rebuild path vs incremental-refresh delegation), not a
   user-intent distinction. [M4-R1](M4-R1.md#1-add-is-the-least-coherent-public-command)
   already established this for the CLI; the same reasoning applies to the
   library.
3. The one semantically meaningful difference — `add()` preserves existing
   traversal configuration while `sync()` always clobbers it — is a
   config-persistence policy that should be folded into the unified `sync()`
   rather than kept as a reason for a separate method. [M4-R1](M4-R1.md#definitive-proposal)
   item 4 already recommends this.

The library change depends on the CLI simplification amendment tracked in
[FW-10](../future-work.md#fw-10-cli-simplification-amendment). It should be
implemented together with the CLI rename, not independently.

#### 4. `remove_root()` naming should follow the CLI rename

[M4-R1](M4-R1.md) recommends renaming the CLI token from `remove-root` to
`remove`. For library/CLI term alignment, the library method should be renamed
from `remove_root()` to `remove()`.

The current name `remove_root()` is more explicit about what is being removed
(a root, not a ticket file), but that specificity is redundant when the class
is already named `ContextSync` and the only removable target is a root. The
method docstring and parameter name make the semantics clear. `remove()` is
shorter, matches the CLI, and reads naturally: `ctx.remove(key="ACP-123")`.

#### 5. Method-level docstrings expose implementation strategy

The `sync()` and `refresh()` docstrings lead with implementation descriptions
rather than caller intent:

| Method | Current first line | Recommended |
| --- | --- | --- |
| `sync()` | "Full-snapshot rebuild from *root_ticket_id* and all existing roots." | "Track a ticket and update the snapshot from all roots." |
| `refresh()` | "Incremental whole-snapshot update from all existing roots." | "Update the snapshot from all tracked roots." |

These are the same implementation-oriented descriptions that [M4-R1](M4-R1.md)
identified in the CLI help text. The library docstrings should describe what
the caller gets, not how the library achieves it. The detailed implementation
notes that follow in the docstring body are fine — the problem is only with the
leading summary line.

Additionally, `SyncResult` at
[src/context_sync/_models.py:42](../../src/context_sync/_models.py#L42)
describes itself as "Outcome of a ``sync``, ``refresh``, ``add``, or
``remove_root`` call." If `add()` is removed from the public surface, this
docstring needs updating.

#### 6. `context_dir` property docstring uses banned terminology

The `context_dir` property docstring at
[src/context_sync/_sync.py:270](../../src/context_sync/_sync.py#L270) reads
"The context directory this syncer operates on." Per
[docs/policies/terminology.md](../policies/terminology.md), this should be
reworded to avoid "syncer" — for example, "The context directory this instance
operates on."

#### 7. Exception hierarchy is sound

The exception hierarchy in
[src/context_sync/_errors.py](../../src/context_sync/_errors.py) is
well-structured:

- Clear base (`ContextSyncError`) with meaningful subtypes for each failure
  category.
- Lock, workspace, root, remote, and I/O errors are distinct and catchable
  individually.
- Docstrings explain when each exception is raised and what the caller should
  do.
- `RootNotInManifestError` is slightly verbose compared to siblings but
  communicates the right thing: the target is tracked but not in the root set.
  No rename recommended.

No exception naming changes are needed.

#### 8. Result model `status` and `error_type` fields are bare strings

`DiffEntry.status` at
[src/context_sync/_models.py:81](../../src/context_sync/_models.py#L81) accepts
one of four values (`"current"`, `"stale"`, `"missing_locally"`,
`"missing_remotely"`) but is typed as `str`. Similarly, `SyncError.error_type`
at [src/context_sync/_models.py:25](../../src/context_sync/_models.py#L25)
accepts values like `"not_found"`, `"permission_denied"`, `"api_error"`,
`"root_quarantined"` as a bare `str`.

A `StrEnum` or `Literal` type would make the valid values discoverable for
callers without reading the docstring. This is a modest ergonomics improvement,
not a blocking issue:

- `DiffEntry.status` has a closed set of four values → `Literal` or `StrEnum`.
- `SyncError.error_type` has an open-ended set that may grow → bare `str` is
  defensible, but documenting the current categories in a `StrEnum` or a
  class-level constant is still worthwhile.

This is a documentation-and-typing improvement that can ride with the M4-3
rename or be deferred separately.

#### 9. Public surface width is large but structured

The `__all__` in
[src/context_sync/__init__.py](../../src/context_sync/__init__.py) exports ~55
symbols across 13 categories. For a typical library caller, the essential
surface is:

- `ContextSync` (5 methods, 4 properties)
- `SyncResult`, `DiffResult`, `DiffEntry`, `SyncError` (result types)
- The `ContextSyncError` hierarchy (12 exception types)
- Configuration constants and `Dimension`

The remaining exports — gateway protocol and data types, manifest types, lock
types, pipeline utilities, I/O utilities, traversal types, renderer, and
signature functions — serve framework consumers and test authors. They are
well-organized with `__all__` comments and do not pollute the primary import
namespace when callers use `from context_sync import ContextSync`.

No change is recommended for 0.x. For 1.0.0, the plan should consider whether
the full 55-symbol `__all__` is the intended stable public surface or whether
a narrower core + an `internal` or `testing` sub-namespace would better
communicate stability guarantees.

### M4-3 assessment

[M4-3](../implementation-plan.md#m4-3---rename-root-ticket-id-to-key) as
currently scoped ("rename `root_ticket_id` to `key` on `sync()` and update
CLI/docstrings/tests") is **too narrow** to address the full naming problem.

The rename should be broadened to cover:

1. `sync(root_ticket_id=...)` → `sync(key=...)` (already in M4-3)
2. `add(ticket_ref=...)` → `add(key=...)` (or `_add` if removed from public
   surface)
3. `remove_root(ticket_ref=...)` → `remove(key=...)` (combining the rename
   with the method rename)
4. `SyncError.ticket_id` → `SyncError.ticket_key`
5. `DiffEntry.ticket_id` → `DiffEntry.ticket_key`

Whether this is done by widening M4-3 or by replacing M4-3 with a broader
"API terminology cleanup" ticket depends on planning preference. The critical
point is that a narrow rename of only `sync(root_ticket_id=...)` would leave
the adjacent naming inconsistencies in place and require a second pass.

### Definitive proposal

1. **Widen M4-3** (or replace with a broader ticket) to cover:
   - `sync(root_ticket_id=...)` → `sync(key=...)`
   - `SyncError.ticket_id` → `SyncError.ticket_key`
   - `DiffEntry.ticket_id` → `DiffEntry.ticket_key`
   - Unify the root-input parameter name to `key` across `add()` and
     `remove_root()` (items 2–3 above).
2. **When the CLI simplification lands** (tracked in
   [FW-10](../future-work.md#fw-10-cli-simplification-amendment)):
   - Make `add()` internal or remove it from the public API.
   - Rename `remove_root()` to `remove()`.
   - Update `sync()` to preserve existing traversal configuration when no
     overrides are supplied (adopting `add()`'s preservation semantics).
3. **Documentation-only changes** that can land without a plan amendment:
   - Reword the `sync()` and `refresh()` docstring summary lines to describe
     caller intent rather than implementation strategy.
   - Update `SyncResult`'s docstring if `add()` is removed.
   - Fix the `context_dir` property docstring to remove the banned
     "syncer" term.
4. **Optional typing improvement** (can ride with M4-3 or be deferred):
   - Add a `Literal` type or `StrEnum` for `DiffEntry.status`.
   - Document `SyncError.error_type` categories more explicitly.
5. **For 1.0.0 planning**: evaluate whether the full 55-symbol `__all__` is
   the intended stable surface or whether a narrower public API with internal
   sub-namespaces is appropriate.

### Follow-on tracking

The API changes recommended here fall into three routing categories:

1. **M4-3 widening**: Items 1.a–1.d of the definitive proposal should be
   handled by amending M4-3's scope in the implementation plan. No new ticket
   is needed; the existing ticket just needs broader acceptance criteria.

2. **CLI simplification coupling**: Items 2.a–2.c are consequences of the CLI
   simplification already tracked in
   [FW-10](../future-work.md#fw-10-cli-simplification-amendment). The FW-10
   scope should be updated to include the library-side method boundary changes
   (make `add()` internal, rename `remove_root()` to `remove()`, adopt
   traversal-config preservation). No new ticket is needed.

3. **Documentation-only**: Items 3.a–3.c can be applied as part of M4-3
   implementation or independently. No plan amendment required.

4. **1.0.0 surface width**: Item 5 is a planning consideration for Milestone 6
   and does not need tracking now.
