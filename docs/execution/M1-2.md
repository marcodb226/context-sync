## Coding Guidelines Compliance

Checked against
[docs/policies/common/coding-guidelines.md](../policies/common/coding-guidelines.md)
and
[docs/policies/common/python/coding-guidelines.md](../policies/common/python/coding-guidelines.md):

- **Documentation**: Every new module has a header docstring. All public
  functions and classes have docstrings with parameter/return/raises
  documentation.
- **No magic numbers**: `FORMAT_VERSION`, `MANIFEST_FILENAME`,
  `LOCK_FILENAME`, `SIGNATURE_PREFIX`, `CRITICAL_FRONTMATTER_FIELDS` all
  named constants.
- **Type annotations**: All function signatures and class fields annotated.
  Pydantic models use `ConfigDict(extra="forbid")`.
- **Data modeling**: Pydantic for all models crossing the serialization
  boundary (manifest, lock). Frozen dataclasses from
  [M1-1](../implementation-plan.md#m1-1---project-scaffold-and-public-runtime-contracts)
  used as inputs.
- **Fail loudly**: `ManifestError` on corrupt/missing/incompatible manifest.
  `ActiveLockError` / `StaleLockError` on lock contention. `WriteError` on
  failed writes or verification mismatches.
- **Atomic writes**: All file writes use temp + rename. No partial files.
- **DRY**: `resolve_root_comment` shared between `_renderer.py` and
  `_testing.py`. YAML normalization shared between manifest and frontmatter.
- **No hardcoded secrets**: Lock uses `platform.node()` and `os.getpid()`;
  no credentials in any persistence file.
- **Stable-release changelog gate**: No entry required — pre-`1.0.0`
  repository.

## Completion Summary

- Deliverables produced:
  - Manifest schema with Pydantic validation, round-trip YAML I/O, and

  format clean.
