## Reviewer Notes

### What the ticket got right

The implementation cleanly addresses both deferred review findings from [M4.1-1-review.md](M4.1-1-review.md):

- **[M4.1-1-R1](M4.1-1-review.md) consolidation:** All three behaviors formerly
  isolated in `_add_under_lock` — alias-based local resolution via
  `_normalize_ticket_ref` + `_resolve_ref_to_uuid`, early URL workspace slug
  validation (fail-fast before any remote call), and quarantine-recovery
  detection with logging — are now consolidated into `_sync_under_lock`
  ([src/context_sync/_sync.py:410-482](../../src/context_sync/_sync.py#L410-L482)).
  The consolidation is well-structured: local resolution happens before the
  fetch, slug validation fails fast, and quarantine recovery logs
  appropriately.

- **[M4.1-1-R2](M4.1-1-review.md) full-rebuild:** `_standalone_sync_under_lock`
  is rewritten from a one-line delegation to `_refresh_under_lock` into a
  genuine full-rebuild pipeline
  ([src/context_sync/_sync.py:562-678](../../src/context_sync/_sync.py#L562-L678)).
  It pre-fetches all active root bundles, builds the reachable graph, then
  delegates to `_sync_write_pass` which fetches all remaining reachable
  tickets unconditionally. The key behavioral distinction from `refresh` is
  now real: `sync` re-fetches everything at the network level; `refresh` only
  fetches tickets identified as stale by cursor comparison.

- **`_sync_write_pass` extraction:** The shared post-traversal pipeline
  ([src/context_sync/_sync.py:680-821](../../src/context_sync/_sync.py#L680-L821))
  is a good DRY improvement. The duplicated fetch-remaining + write + prune +
  finalize sequence is extracted into a single method used by both sync paths,
  with the `log_label` parameter providing caller-specific log context.

- **Dead code removal:** `_add()` and `_add_under_lock()` are fully deleted
  (~170 lines), and the `"add"` mode is cleanly removed from `LOCK_MODES`
  ([src/context_sync/_lock.py:33](../../src/context_sync/_lock.py#L33)) and
  `ManifestSnapshot.mode`
  ([src/context_sync/_manifest.py:90](../../src/context_sync/_manifest.py#L90)).
  Associated docstrings are updated throughout.

- **Test migration:** All 15 `._add()` callsites in
  [tests/test_add_remove_root.py](../../tests/test_add_remove_root.py) are
  rewritten to `.sync()`. Class and method names are appropriately renamed.
  Snapshot mode assertions correctly changed from `"add"` to `"sync"`.
  [tests/test_package.py](../../tests/test_package.py) correctly asserts
  neither `add` nor `_add` exist on `ContextSync`.

- **Validations:** All declared validations passed — ruff check, ruff format,
  469 pytest tests, and cloc confirms
  [src/context_sync/_sync.py](../../src/context_sync/_sync.py) at 972 code
  lines (under the 1,000-line limit).

### Residual risks

- **Standalone sync full-rebuild behavior is untested at the library level.**
  The core behavioral change this ticket delivers — standalone `sync` performs
  a full rebuild rather than delegating to the incremental refresh path — has
  no regression guard beyond a CLI exit-code check. If a future refactor
  reverts standalone sync to incremental behavior (the exact bug this ticket
  fixes), no test would fail.

### Testing gaps

- No test for standalone `sync` with multiple roots where one root is
  unavailable during pre-fetch — the warning/exclusion path in
  `_standalone_sync_under_lock` ([src/context_sync/_sync.py:622-627](../../src/context_sync/_sync.py#L622-L627))
  is not exercised.
- No test for standalone `sync` with explicit dimension/cap overrides —
  the override application path
  ([src/context_sync/_sync.py:594-597](../../src/context_sync/_sync.py#L594-L597))
  is not exercised. (This gap was also noted in the
  [M4.1-1 review](M4.1-1-review.md) and remains.)
