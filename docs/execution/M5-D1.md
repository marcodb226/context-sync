- Confirmed the two major v1.0.0 gaps that v1.1.0 closes for ordinary ticket fetches: per-issue comment metadata is now domain-layer (`Comment.updated_at`, `resolved_at`, `parent`, `children`) and per-issue relation reads are now domain-layer (`Issue.get_links()` / `IssueLink`). The new audit therefore keeps those reads on packaged surfaces and does not preserve the old raw-only conclusion for single-ticket fetches.
- Confirmed that the packaged surface still lacks stable workspace identity for a fetched issue. The installed `Linear.me()` viewer query still returns only `id`, `name`, and `email`, and no inspected issue/team/user path exposes workspace id plus slug. The new audit therefore preserves a narrow raw helper for workspace binding.
- Confirmed that the packaged surface still lacks a read path for the current issue label set. `context-sync` still persists normalized label strings in ticket frontmatter, but `Issue.get_labels()` in [linear-client src/linear_client/domain/issue.py](../../../linear-client/src/linear_client/domain/issue.py) explicitly documents that labels are not available from the current issue payload. The new audit records this as an additional required helper for ticket rendering.
- Confirmed that the packaged surface still lacks the by-id batch reads required by the refresh design: issue `updated_at` by tracked id set, comment/thread metadata by tracked issue set, and relation metadata by tracked issue set. The new audit keeps those three batched refresh helpers inside the raw boundary.
- Chose the narrower v1.1.0 boundary recorded in [docs/design/linear-domain-coverage-audit-v1.1.0.md](../design/linear-domain-coverage-audit-v1.1.0.md): use the domain layer for ordinary issue, comment, attachment, and per-issue relation reads; keep raw GraphQL only for workspace identity, single-issue label rendering, and the three batched refresh-metadata helpers.
- Recorded that `get_ticket_relations(issue_ids)` remains a gateway helper but is no longer part of the approved raw-helper set. Its approved implementation path is domain-layer fan-out over `Issue.get_links()` with deduplication and bounded concurrency owned by the gateway implementation.
- Recorded the type-surface reconciliation outcome in the new audit: the five exact duplicate aliases (`IssueId`, `IssueKey`, `CommentId`, `AttachmentId`, and `Timestamp`/`IsoTimestamp`) should share one `NewType` identity by re-exporting upstream aliases from [src/context_sync/_types.py](../../src/context_sync/_types.py); `WorkspaceId`, `WorkspaceSlug`, and `WriterId` remain local; `AssetUrl` and `IssueLinkType` should be adopted at the gateway boundary where the current dataclasses still use bare `str`; and the remaining display-oriented fields should stay plain `str`.
- Evaluated the optional v1.1.0 metadata enrichments (`Status.category`, `Issue.team`, `Issue.delegate`, `User.app`) against the current v1 snapshot contract in [docs/adr.md](../adr.md#2-persistence-format) and [src/context_sync/_renderer.py](../../src/context_sync/_renderer.py). The audit explicitly keeps them out of the current boundary because the persisted ticket contract still stores status name, assignee/creator display names, labels, attachments, and relations only. Adding team, delegate, or workflow-category output would widen the user-facing snapshot contract and must enter through a future amendment rather than through opportunistic gateway implementation.
- Confirmed that the implementation consequences discovered by the audit already have an execution destination in [M5-1](../implementation-plan.md#m5-1---real-linear-gateway-and-runtime-wiring): the ticket notes already require a single `NewType` identity across the boundary and assign the real gateway responsibility for all required ticket-bundle and refresh-metadata reads. No new follow-on ticket or plan amendment was needed from this Phase A pass.
