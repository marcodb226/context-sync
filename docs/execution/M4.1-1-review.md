## Reviewer Notes

### What the ticket got right

The core structural changes are clean and well-executed:

- The CLI parser correctly makes `sync`'s TICKET optional with `nargs="?"`.
- The `_build_dimensions` helper correctly returns `None` when no `--depth-*` flags
  are given, and the CLI defaults `--max-tickets-per-root` to `None`. This gives
  the library layer the signal it needs to preserve manifest config.
- `_sync_under_lock` correctly preserves manifest config when no override is
  supplied ([src/context_sync/_sync.py:433-441](../../src/context_sync/_sync.py#L433-L441)),
  and correctly uses instance defaults only for new-manifest initialization.
- The `remove-root` → `remove` rename is thorough and consistent across CLI,
  library, lock, manifest, and tests.
- The `add` CLI subcommand and handler are fully removed.
- `_add()` is correctly underscore-prefixed and absent from
  [src/context_sync/__init__.py `__all__`](../../src/context_sync/__init__.py).
- [README.md](../../README.md) documentation is comprehensive, with all six use
  cases and updated options tables.
- Test renames and assertion updates are consistent with the behavioral changes.
- All declared validations passed (ruff, formatting, pytest, cloc).

### Residual risks

- **Dual root-addition code paths:**
  [`_sync_under_lock`](../../src/context_sync/_sync.py#L387) (used by
  `sync TICKET`) and
  [`_add_under_lock`](../../src/context_sync/_sync.py#L1302) (used only by
  tests via `_add`) implement root addition with different flows.
  `_add_under_lock` has three behaviors absent from `_sync_under_lock`: (1)
  alias-based local resolution via `_normalize_ticket_ref` +
  `_resolve_ref_to_uuid`, avoiding redundant remote round-trips when the key is
  already known locally; (2) early URL workspace slug validation that rejects
  mismatched slugs before any network call; (3) explicit quarantine-recovery
  detection and logging when a previously quarantined root returns to active
  state. These behaviors are tested (~30 callsites in
  [tests/test_add_remove_root.py](../../tests/test_add_remove_root.py) and
  [tests/test_e2e.py](../../tests/test_e2e.py)) but unreachable from any public
  API path. The recommended resolution (M4.1-1-R1) is to consolidate these
  behaviors into `_sync_under_lock`, delete `_add` entirely, and track the work
  as a separate implementation ticket on the current milestone without a change
  request.

- **Standalone sync / refresh equivalence:** Until M4.1-1-R2 is addressed, the
  documented behavioral distinction between standalone `sync` and `refresh` does
  not exist in the implementation. Users cannot rely on `sync` to force a full
  re-fetch to repair corrupted or stale local state.

### Testing gaps

Beyond the two explicitly missing tests (M4.1-1-R3, M4.1-1-R4), the following
testing scenarios have no direct coverage:

- Standalone `sync` on an empty context directory (no manifest) — the
  [`_standalone_sync_under_lock`](../../src/context_sync/_sync.py#L640) path
  calls `load_manifest` which should raise `ManifestError`, and the `sync()`
  docstring documents this. No test exercises this path.
- Standalone `sync` with explicit dimension/cap overrides — the override
  application branch in
  [`_standalone_sync_under_lock`](../../src/context_sync/_sync.py#L666-L669) is
  not exercised by any test.
- `sync TICKET` called a second time with no overrides on an existing manifest
  where the manifest already has non-default config — this is the preservation
  path in
  [`_sync_under_lock`](../../src/context_sync/_sync.py#L433-L441).
