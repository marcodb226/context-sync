ADR-R1. The updated_at Freshness Gamble
The current incremental update strategy rests on a potentially brittle foundation regarding how the upstream API handles data mutations.

The refresh flow relies on a batched GraphQL query to check ticket updated_at values against local last_synced_at timestamps.

The architecture explicitly acknowledges the need to validate if issue-level updated_at actually advances when child entities, like comments, are added or edited.

If this validation fails, the design states the refresh mechanism must be revised.

The Weakness: In many complex GraphQL APIs, mutating an edge (like appending a comment, adding an attachment, or linking a relation) does not reliably bump the top-level parent node's updated_at timestamp. If Linear's schema behaves this way, the refresh mode will silently ignore new comments or changed blockers, feeding agents stale context.

The Fix: Do not rely solely on the top-level issue timestamp. The batched GraphQL freshness query must explicitly request the updatedAt fields for the relevant sub-edges (e.g., comments(last: 1) { nodes { updatedAt } }). Alternatively, you may need to leverage Linear's synchronization API payloads directly to compute accurate deltas.

ADR-R2. Lock Files and Orchestrator Safety
The deferral of whole-snapshot atomic commits creates a high risk of deadlocks and poisoned context in automated environments.

Mutating operations utilize a transient .context-sync.lock file to prevent concurrent writers.

If a writer is active, subsequent diff or sync attempts will fail fast with a clear error rather than waiting.

If a run is interrupted, the directory may be left in a partially updated state.

Stronger atomic commit semantics are deferred to future work.

The Weakness: Asynchronous workers frequently experience retries, timeouts, or hard terminations. If a worker is killed mid-run, it leaves behind a stale .context-sync.lock file and a corrupted context directory. Because the tool is designed to fail fast when it sees a lock, all future automated retries will instantly fail until a human manually deletes the lock file. Furthermore, a partially written directory will feed fractured graph data to the next agent that reads it.

The Fix: Pull FW-3 into the first release. Write the new snapshot to a staging directory (e.g., .context-sync.tmp/) and use an atomic OS-level rename to swap it with the live directory upon success. Embed the writer's PID and a timestamp inside the lock file to allow subsequent runs to identify and preempt demonstrably stale locks.

ADR-R3. Pure Breadth-First Traversal Cutoffs
The graph traversal strategy may unintentionally omit critical structural dependencies when operating near its safety boundaries.

Traversal explores the graph breadth-first from all roots simultaneously.

If the maximum ticket count (default 200) is reached mid-traversal, the system stops and returns the collected subset.

The Weakness: With a pure breadth-first approach across multiple roots, the engine treats all edge types equally at a given depth. If you have 20 roots, you might hit the 200-ticket cap simply by fetching loose relates_to tickets for the first 15 roots. This could result in the tool failing to fetch a critical blocks ticket for the 16th root, blinding the agent to a hard dependency.

The Fix: Implement a dimension-weighted priority queue for the traversal rather than a strictly chronological BFS. Weight the queue so that structural edges (like blocks and parent) are always processed before informational edges (like relates_to or ticket_ref), regardless of which root they originate from.

ADR-R4. Terminal Root Fragility
The failure model for root tickets is too aggressive for long-running context directories.

If an existing manifest root is unavailable during a refresh operation, the run fails immediately.

The tool intentionally does not distinguish between a ticket being permanently deleted and a ticket being missing due to lost permissions.

The Weakness: If a human deletes a single stale issue in Linear that happens to be tracked as a root in a local directory, the entire refresh process for that directory will permanently hard-fail. The agent loop will completely break down because one ticket out of potentially hundreds disappeared upstream.

The Fix: Do not make root unavailability a terminal run failure during refresh. Instead, introduce a "quarantined" state in the manifest. If a root cannot be fetched, flag it as quarantined, skip its graph traversal, and proceed with syncing the healthy roots. Expose the quarantined status in the errors list of the SyncResult so the calling orchestrator can decide whether to alert an operator or automatically invoke remove_root.

ADR-R5. The "Thundering Herd" Rate Limit Risk
Delegating rate limit handling entirely to the underlying HTTP client is risky when running inside a distributed workflow orchestrator.

The design states that the tool relies on linear-client for retry and backoff, explicitly avoiding cross-process rate-limit coordination.

This means if your orchestrator spins up 50 concurrent activity workers handling different contexts, they will all independently hit the Linear API, exhaust the global workspace limit, and blindly sleep/retry in memory.

The Flaw: In an asynchronous orchestration system, having dozens of workers sleeping in memory to handle 429 backoffs consumes valuable worker capacity and risks activity timeouts.

The Fix: The library API should accept an optional backoff/yield callback. Instead of forcing the Python asyncio loop to sleep internally, the tool can yield a rate-limit exception back to the orchestrator, allowing the orchestrator to schedule the retry and free up the worker thread.

ADR-R6. File Renaming Breaks Strict Idempotency
The decision to use mutable issue keys for filenames creates race conditions and complicates recovery for interrupted workers.

Stable ticket identity relies on the immutable Linear UUID, but the tool names the actual Markdown files using the current human-facing issue key (e.g., <current-ticket-key>.md).

If a ticket key changes upstream, the tool renames the local file, updates the manifest, and records the old key as an alias.

The Flaw: If a worker crashes or times out after renaming the file on disk but before writing the updated .context-sync.yml manifest, the state is permanently corrupted. The next idempotent retry will fail to find the file matching the old manifest mapping.

The Fix: Name the physical files using the immutable Linear UUID (e.g., 9c9e3d7a-....md). Use the .context-sync.yml manifest entirely for alias resolution, or generate lightweight symlinks for the human-readable issue keys. This ensures file paths are append-only/overwrite-in-place and never require coordinated rename operations.

ADR-R7. Memory Exhaustion via Unbounded Comment Payloads
The safety limits currently focus entirely on graph width and depth, ignoring payload depth.

The traversal model enforces a strict maximum ticket-count cap, defaulting to 200.

However, the design also dictates that the first release must include the full comment history returned by Linear in every ticket file.

The Flaw: A single incident-response or debate-heavy ticket can accumulate hundreds of comments. Fetching 200 tickets that each contain 100 comments will result in massive GraphQL payloads. This risks Out-Of-Memory (OOM) errors in the worker process during the deterministic rendering phase and bloats the context directory beyond what an agent model can comfortably ingest.

The Fix: Do not defer comment truncation entirely to future work. Implement a hard character or item limit per comment thread in version 1 (e.g., fetching only the last 50 comments per ticket), or add a max_comments_per_ticket safety bound alongside the max_tickets configuration.
