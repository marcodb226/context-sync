# ADR Review

This review highlights the primary design risks identified in the ADR and outlines recommended changes before implementation proceeds.

Status labels in parentheses reflect the current tracking state for each review item.

## ADR-R1 (Discarded). The `updated_at` Freshness Gamble

The current incremental update strategy rests on a potentially brittle foundation regarding how the upstream API handles data mutations.

### Current design

- The refresh flow relies on a batched GraphQL query to check ticket `updated_at` values against local `last_synced_at` timestamps.
- The architecture explicitly acknowledges the need to validate whether issue-level `updated_at` actually advances when child entities, such as comments, are added or edited.
- If this validation fails, the design states the refresh mechanism must be revised.

### Weakness

In many complex GraphQL APIs, mutating an edge, such as appending a comment, adding an attachment, or linking a relation, does not reliably bump the top-level parent node's `updated_at` timestamp. If Linear's schema behaves this way, the refresh mode will silently ignore new comments or changed blockers, feeding agents stale context.

### Fix

Do not rely solely on the top-level issue timestamp. The batched GraphQL freshness query must explicitly request the `updatedAt` fields for the relevant sub-edges, for example `comments(last: 1) { nodes { updatedAt } }`. Alternatively, you may need to leverage Linear's synchronization API payloads directly to compute accurate deltas.

## ADR-R2a (Done). Stale Lock Recovery and Lock Metadata

The current writer-lock design is too brittle for automated retries after interruption.

### Current design

- Mutating operations utilize a transient `.context-sync.lock` file to prevent concurrent writers.
- If a writer is active, subsequent diff or sync attempts will fail fast with a clear error rather than waiting.

### Weakness

Asynchronous workers frequently experience retries, timeouts, or hard terminations. If a worker is killed mid-run, it leaves behind a stale `.context-sync.lock` file. Because the tool is designed to fail fast when it sees a lock, all future automated retries will instantly fail until a human manually deletes the lock file.

### Fix

Embed the writer's PID and a timestamp inside the lock file so subsequent runs can identify and preempt demonstrably stale locks.

## ADR-R2b (Postponed: FW-3). Whole-Snapshot Atomic Commit

The first-release snapshot write path remains vulnerable to interrupted runs.

This review item does not ask for anything beyond [FW-3](../future-work.md#fw-3-whole-snapshot-atomic-commit); it is the same deferred work stated in review form.

### Current design

- If a run is interrupted, the directory may be left in a partially updated state.
- Stronger atomic commit semantics are deferred to future work.

### Weakness

If a run dies after updating only part of the context directory, the next reader will see a fractured snapshot assembled from two different passes.

### Fix

Pull [FW-3](../future-work.md#fw-3-whole-snapshot-atomic-commit) into the first release. Write the new snapshot to a staging directory, for example `.context-sync.tmp/`, and use an atomic OS-level rename to swap it with the live directory upon success.

## ADR-R3 (Done). Pure Breadth-First Traversal Cutoffs

The graph traversal strategy may unintentionally omit critical structural dependencies when operating near its safety boundaries.

### Current design

- Traversal explores the graph breadth-first from all roots simultaneously.
- If the maximum ticket count, default `200`, is reached mid-traversal, the system stops and returns the collected subset.

### Weakness

With a pure breadth-first approach across multiple roots, the engine treats all edge types equally at a given depth. If you have 20 roots, you might hit the 200-ticket cap simply by fetching loose `relates_to` tickets for the first 15 roots. This could result in the tool failing to fetch a critical `blocks` ticket for the 16th root, blinding the agent to a hard dependency.

### Fix

Implement a dimension-weighted priority queue for the traversal rather than a strictly chronological BFS. Weight the queue so that structural edges, such as `blocks` and `parent`, are always processed before informational edges, such as `relates_to` or `ticket_ref`, regardless of which root they originate from.

## ADR-R4 (Done). Terminal Root Fragility

The failure model for root tickets is too aggressive for long-running context directories.

### Current design

- If an existing manifest root is unavailable during a refresh operation, the run fails immediately.
- The tool intentionally does not distinguish between a ticket being permanently deleted and a ticket being missing due to lost permissions.

### Weakness

If a human deletes a single stale issue in Linear that happens to be tracked as a root in a local directory, the entire refresh process for that directory will permanently hard-fail. The agent loop will completely break down because one ticket out of potentially hundreds disappeared upstream.

### Fix

Do not make root unavailability a terminal run failure during refresh. Instead, introduce a "quarantined" state in the manifest. If a root cannot be fetched, flag it as quarantined, skip its graph traversal, and proceed with syncing the healthy roots. Expose the quarantined status in the errors list of the `SyncResult` so the calling orchestrator can decide whether to alert an operator or automatically invoke `remove_root`.

## ADR-R5 (Discarded). The "Thundering Herd" Rate Limit Risk

Delegating rate limit handling entirely to the underlying HTTP client is risky when running inside a distributed workflow orchestrator.

### Current design

- The design states that the tool relies on `linear-client` for retry and backoff, explicitly avoiding cross-process rate-limit coordination.
- This means if your orchestrator spins up 50 concurrent activity workers handling different contexts, they will all independently hit the Linear API, exhaust the global workspace limit, and blindly sleep or retry in memory.

### Flaw

In an asynchronous orchestration system, having dozens of workers sleeping in memory to handle `429` backoffs consumes valuable worker capacity and risks activity timeouts.

### Fix

The library API should accept an optional backoff or yield callback. Instead of forcing the Python `asyncio` loop to sleep internally, the tool can yield a rate-limit exception back to the orchestrator, allowing the orchestrator to schedule the retry and free up the worker thread.

## ADR-R6 (Discarded). File Renaming Breaks Strict Idempotency

The decision to use mutable issue keys for filenames creates race conditions and complicates recovery for interrupted workers.

### Current design

- Stable ticket identity relies on the immutable Linear UUID, but the tool names the actual Markdown files using the current human-facing issue key, for example `<current-ticket-key>.md`.
- If a ticket key changes upstream, the tool renames the local file, updates the manifest, and records the old key as an alias.

### Flaw

If a worker crashes or times out after renaming the file on disk but before writing the updated `.context-sync.yml` manifest, the state is permanently corrupted. The next idempotent retry will fail to find the file matching the old manifest mapping.

### Fix

Name the physical files using the immutable Linear UUID, for example `9c9e3d7a-....md`. Use the `.context-sync.yml` manifest entirely for alias resolution, or generate lightweight symlinks for the human-readable issue keys. This ensures file paths are append-only or overwrite-in-place and never require coordinated rename operations.

## ADR-R7 (Postponed). Memory Exhaustion via Unbounded Comment Payloads

The safety limits currently focus entirely on graph width and depth, ignoring payload depth.

### Current design

- The traversal model enforces a strict maximum ticket-count cap, defaulting to `200`.
- However, the design also dictates that the first release must include the full comment history returned by Linear in every ticket file.

### Flaw

A single incident-response or debate-heavy ticket can accumulate hundreds of comments. Fetching 200 tickets that each contain 100 comments will result in massive GraphQL payloads. This risks Out-Of-Memory (OOM) errors in the worker process during the deterministic rendering phase and bloats the context directory beyond what an agent model can comfortably ingest.

### Fix

Do not defer comment truncation entirely to future work. Implement a hard character or item limit per comment thread in version 1, for example fetching only the last 50 comments per ticket, or add a `max_comments_per_ticket` safety bound alongside the `max_tickets` configuration.

I would not hardcode a more complex fallback query shape until that validation proves it is necessary. |
| ADR-R2a | Do now | Lock hygiene should be strengthened now by recording enough metadata to detect and clear stale locks safely. |
| ADR-R2b | Defer to [FW-3](../future-work.md#fw-3-whole-snapshot-atomic-commit) | I would keep whole-snapshot staging and atomic directory swap deferred to [FW-3](../future-work.md#fw-3-whole-snapshot-atomic-commit). This review item does not introduce any extra scope beyond that existing future-work item. |
| ADR-R3 | Do now | The underlying risk is valid. I would amend the traversal design so structural dimensions are prioritized ahead of informational ones when the traversal approaches the ticket cap. I would not commit to the review's exact weighted-queue implementation yet, but I would change the ADR so blockers, parent/child structure, and similar dependency edges win over `relates_to` or `ticket_ref` near the bound. |
| ADR-R4 | Do now | I would change `refresh` semantics for already tracked roots so an unavailable root becomes quarantined by default rather than making the entire refresh terminal, while still allowing an explicit refresh option to remove that root immediately when the caller wants destructive cleanup. I would keep `sync` and `add` strict for explicitly requested roots, because those operations still need a hard failure if the requested ticket is not available in the caller's visible workspace state. |
| ADR-R5 | Invalid | I would not change the ADR on this point. Cross-process rate-limit coordination and orchestrator-yield semantics belong either in `linear-client` or in the embedding runtime, not in the v1 `context-sync` design. If later operating experience shows the current layering is insufficient, that should become a follow-up against the integration boundary rather than a blocker for this ADR. |
| ADR-R6 | Invalid | I would keep human-readable issue-key filenames. Readability and direct browseability are part of the persistence-format value proposition, and UUID filenames would weaken that for an interrupted-run scenario that is better addressed through write ordering, recovery logic, or [FW-3](../future-work.md#fw-3-whole-snapshot-atomic-commit). |
| ADR-R7 | Defer to [FW-1](../future-work.md#fw-1-comment-storage-optimizations) | I would keep the v1 correctness contract of storing full comment history and defer truncation or per-ticket comment caps to [FW-1](../future-work.md#fw-1-comment-storage-optimizations). The right near-term mitigation is observability during implementation, such as tracking comment counts and rendered artifact size, rather than weakening snapshot completeness before we have evidence that payload size is a real operational problem. |

The concrete ADR changes I would make now are:

- strengthen stale-lock handling from ADR-R2a without pulling all of [FW-3](../future-work.md#fw-3-whole-snapshot-atomic-commit) into v1;
- keep ADR-R2b postponed to [FW-3](../future-work.md#fw-3-whole-snapshot-atomic-commit), with no extra scope beyond that item;
- prioritize structural edges near the traversal cap as described in ADR-R3;
- quarantine unavailable existing roots during `refresh` by default, with an explicit destructive option to remove them immediately, as described in ADR-R4.
