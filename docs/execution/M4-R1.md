### Review conclusions

#### 1. `add` is the least coherent public command

The strongest simplification available without rewriting the whole CLI model is
to remove `add` from the public surface.

The current surface asks operators to choose between `sync <ticket>` and
`add <ticket>` for the same visible outcome: "make this ticket part of the
tracked root set and update the snapshot." The difference is mostly internal:

- `sync` persists traversal overrides such as
  [src/context_sync/_cli.py:403](../../src/context_sync/_cli.py#L403) and
  performs the full-snapshot path in
  [src/context_sync/_sync.py](../../src/context_sync/_sync.py#L290).
- `add` uses the existing manifest configuration and delegates to the refresh
  pipeline in [src/context_sync/_sync.py:1233](../../src/context_sync/_sync.py#L1233).

Those are implementation differences, not user-intent differences. A human
operator deciding "start tracking `TEAM-99`" should not need to choose between
two verbs based on whether the repository will internally take the full rebuild
path or the incremental refresh path.

#### 2. The smaller consistent public set is four commands

The recommended public CLI action set is:

- `sync`
- `refresh`
- `remove-root`
- `diff`

Recommended command meanings:

- `sync <ticket>` becomes the canonical "ensure this root is tracked and
  reconcile the snapshot" command. On an empty context directory it bootstraps
  the snapshot. On an existing directory it adds or reaffirms the supplied root
  and updates the snapshot from the resulting root set.
- `refresh` remains the explicit "update what I already track" command. It does
  not change the root set and does not ask the operator to supply a ticket.
- `remove-root <ticket>` remains the explicit destructive command for dropping
  a root.
- `diff` remains the read-only inspection command.

This keeps the public verbs aligned to operator intent:

- start or expand tracking
- update tracked state
- stop tracking a root
- inspect drift

#### 3. `refresh` should stay separate for now

It would be possible to collapse `refresh` into `sync`, but that would be a
larger semantic rewrite than is needed to eliminate `add`.

Today `refresh` communicates a clean, low-ambiguity meaning: "reconcile the
existing root set with live state." Keeping that verb preserves an explicit
maintenance command while still removing the most confusing overlap from the
surface.

#### 4. `remove-root` should stay explicit for now

The review considered whether root removal should move under a broader `sync`
surface (for example `sync --remove`). I do **not** recommend that.

Removing a root is destructive state mutation. The current explicit verb in
[src/context_sync/_cli.py:429](../../src/context_sync/_cli.py#L429) makes that
fact obvious in a way that a flag on a more general command would not.

#### 5. The CLI simplification has API consequences

If the public CLI removes `add`, then [M4-R2](../implementation-plan.md#m4-r2---api-interface-review)
should explicitly evaluate whether public [ContextSync.add()](../../src/context_sync/_sync.py#L1170)
should remain part of the stable library surface or become an internal helper
behind [ContextSync.sync()](../../src/context_sync/_sync.py#L290).

That also means the currently planned cleanup in
[M4-3](../implementation-plan.md#m4-3---rename-root-ticket-id-to-key) may be
too narrow. The deeper terminology problem is not only `root_ticket_id`; it is
that the current public surface mixes `root_ticket_id`, `root_ticket`, and
`ticket_ref` for closely related concepts.

### Policy-adjacent clarity gaps

No blocking common-policy violation was found in the current flag set, but the
review did identify these consistency gaps:

1. The current CLI surface uses `root_ticket` for `sync` and `ticket_ref` for
   `add` / `remove-root`, even though all three positions accept a human-facing
   issue key or Linear URL. This is a terminology drift problem rather than a
   shared-policy violation.
2. The public library surface still exposes
   [ContextSync.sync(root_ticket_id=...)](../../src/context_sync/_sync.py#L290),
   while [M4-3](../implementation-plan.md#m4-3---rename-root-ticket-id-to-key)
   already acknowledges that `_id` is misleading for a human-facing key or URL.
3. The review conclusions above depend on source inspection and existing test
   artifacts. A live installed-environment CLI smoke path is still owned by
   [M5-2](../implementation-plan.md#m5-2---supported-public-runtime-validation-and-smoke-path),
   not by this review ticket.

### Follow-on tracking required before implementation

The command-surface simplification recommended here is **not** already tracked
as a named active-plan item.

Before implementation, route the change through the planning model as a
material plan amendment that:

1. removes `add` from the public CLI surface and folds its user intent into
   `sync`;
2. defines whether `sync` hides the full-vs-incremental distinction entirely or
   exposes an explicit full-rebuild switch only when configuration changes are
   requested;
3. updates [M4-R2](../implementation-plan.md#m4-r2---api-interface-review) so
   the API review explicitly covers the consequences for
   [ContextSync.add()](../../src/context_sync/_sync.py#L1170) and related
   naming/terminology cleanup;
4. decides whether [M4-3](../implementation-plan.md#m4-3---rename-root-ticket-id-to-key)
   should remain a narrow rename ticket or be replaced by a broader public API
   terminology cleanup.
