### Review conclusions

#### 1. `add` is the least coherent public command

The strongest simplification available without rewriting the whole CLI model is
to remove `add` from the public surface.

The current surface asks operators to choose between `sync <ticket>` and
`add <ticket>` for the same visible outcome: "make this ticket part of the
tracked root set and update the snapshot." The difference is mostly internal:

- `sync` persists traversal overrides such as
  [src/context_sync/_cli.py:403](../../src/context_sync/_cli.py#L403) and
  performs the full-snapshot path in
  [src/context_sync/_sync.py](../../src/context_sync/_sync.py#L290).
- `add` uses the existing manifest configuration and delegates to the refresh
  pipeline in [src/context_sync/_sync.py:1233](../../src/context_sync/_sync.py#L1233).

Those are implementation differences, not user-intent differences. A human
operator deciding "start tracking `TEAM-99`" should not need to choose between
two verbs based on whether the repository will internally take the full rebuild
path or the incremental refresh path.

The public contract should also avoid implying that "adding" a root is a
special narrow update of only the newly supplied ticket's neighborhood. The
snapshot should stay coherent as one tracked state, not as a mental model of
"old tickets from time T plus only the new branch from time T1." The current
[ContextSync.add()](../../src/context_sync/_sync.py#L1170) implementation
already avoids that narrower model by delegating to the shared refresh path
over the active root set; the CLI should not expose a separate verb that makes
operators reason about a distinction the system is trying to hide.

#### 2. The smaller consistent public set is four commands

The recommended public CLI action set is:

- `sync`
- `refresh`
- `remove`
- `diff`

Recommended command meanings:

- `sync <ticket>` becomes the canonical "ensure this root is tracked and
  reconcile the snapshot" command. On an empty context directory it bootstraps
  the snapshot. On an existing directory it adds or reaffirms the supplied root
  and reconciles the snapshot from the resulting root set. The current
  [ContextSync.sync()](../../src/context_sync/_sync.py#L290) implementation
  already performs this whole-set reconciliation: it adds the root to the
  manifest, loads all roots, and recomputes the full reachable graph — so the
  recommended public semantic matches the existing library behavior without
  additional implementation work on the reconciliation path.
- `refresh` remains the explicit "update what I already track" command. It does
  not change the root set and does not ask the operator to supply a ticket.
- `remove <ticket>` remains the explicit destructive command for dropping
  a root.
- `diff` remains the read-only inspection command.

This keeps the public verbs aligned to operator intent:

- start or expand tracking
- update tracked state
- stop tracking a root
- inspect drift

#### 3. `refresh` should stay separate for now

It would be possible to collapse `refresh` into `sync`, but that would be a
larger semantic rewrite than is needed to eliminate `add`.

Today `refresh` communicates a clean, low-ambiguity meaning: "reconcile the
existing root set with live state." Keeping that verb preserves an explicit
maintenance command while still removing the most confusing overlap from the
surface.

#### 3a. Full-vs-incremental execution strategy alternatives

The plan notes explicitly asked for an evaluation of whether the unified `sync`
should be "always incremental when possible" or offer a factored surface such as
`sync --full` for explicit full rebuild. Three alternatives were considered:

| Alternative | Description | Assessment |
| --- | --- | --- |
| **Hidden strategy** (recommended) | The library chooses full rebuild for new roots and incremental refresh for existing roots. The operator never sees the distinction. | This is the simplest operator model. The current [ContextSync.sync()](../../src/context_sync/_sync.py#L290) already uses the full-rebuild path, while [ContextSync.add()](../../src/context_sync/_sync.py#L1170) delegates to the incremental refresh pipeline. A unified `sync` could select internally based on whether the root already exists in the manifest. The operator's mental model is: "`sync TICKET` ensures the ticket is tracked and the snapshot is current." No further reasoning is required. |
| **`sync --full`** | Expose a `--full` flag that forces the full-rebuild path even for existing roots. Default behavior is incremental when possible. | Adds a flag whose value most operators would never need to change. The primary use case — forcing a full rebuild to recover from a suspected inconsistency — is legitimate but rare. The same recovery can be achieved by removing and re-syncing the root (`remove TICKET && sync TICKET`). Adding `--full` increases surface area for a niche recovery scenario. |
| **Current model** (two commands) | Keep `sync` as full rebuild and `add` as incremental. | Already rejected in §1: the distinction is an implementation detail, not a user intent. |

**Recommendation:** adopt the hidden-strategy model. The full-vs-incremental
execution distinction should remain entirely internal. Operators should not see a
`--full` flag. The rare recovery scenario is adequately served by `remove` +
`sync`. This keeps the unified `sync` command simple and avoids exposing
execution-strategy knobs that create the same operator confusion that removing
`add` is meant to eliminate.

#### 4. Root removal should stay explicit, but the token should be shorter

The review considered whether root removal should move under a broader `sync`
surface (for example `sync --remove`). I do **not** recommend that.

Removing a root is destructive state mutation. The current explicit verb in
[src/context_sync/_cli.py:429](../../src/context_sync/_cli.py#L429) makes that
fact obvious in a way that a flag on a more general command would not.

I do recommend renaming the public CLI token from `remove-root` to `remove`.
That keeps the operation explicit while reducing verbosity and bringing it
closer to the direct operator intent.

That CLI rename does not, by itself, require renaming the underlying
[ContextSync.remove_root()](../../src/context_sync/_sync.py#L1333) method. The
library API can keep `remove_root()` unless [M4-R2](../implementation-plan.md#m4-r2---api-interface-review)
concludes that a broader API naming cleanup is worth doing.

The plan notes also asked for an evaluation of `unsync TICKET` as an alternative
verb. I do not recommend `unsync`. The word implies reversing `sync` — that is,
undoing the entire snapshot creation — while the actual operation is narrower:
dropping one root from a potentially multi-root tracked set. `remove` describes
the real semantics more accurately. An operator who reads `unsync TEAM-42` could
reasonably expect that the snapshot for `TEAM-42` is deleted outright; `remove`
communicates that the tool is removing a root from its tracking set, not
destroying the context directory.

#### 5. The CLI simplification has API consequences

If the public CLI removes `add`, the library API surface may also need
adjustment. Whether public
[ContextSync.add()](../../src/context_sync/_sync.py#L1170) should remain part
of the stable library surface or become an internal helper behind
[ContextSync.sync()](../../src/context_sync/_sync.py#L290), and whether
[M4-3](../implementation-plan.md#m4-3---rename-root-ticket-id-to-key) should
remain a narrow rename or become a broader terminology cleanup, are API-scoped
questions owned by
[M4-R2](../implementation-plan.md#m4-r2---api-interface-review). This CLI
review notes the implications but does not make API-shape determinations.

### Definitive proposal

The definitive proposal from this review is:

1. Remove the public `add` CLI action.
2. Rename the public CLI token `remove-root` to `remove`.
3. Make `sync <TICKET>` the only root-expanding command. On an existing
   snapshot, `sync` should reconcile the full tracked set plus the supplied
   ticket, not only the new ticket's neighborhood. The current
   [ContextSync.sync()](../../src/context_sync/_sync.py#L290) already performs
   this reconciliation, so no new library work is needed for this behavior.
4. When `sync` subsumes `add`'s role, a bare `sync TICKET` invoked without
   explicit `--max-tickets-per-root` or `--depth-*` overrides should preserve
   the manifest's existing traversal configuration for that root rather than
   overwriting it with CLI defaults. Today
   [sync](../../src/context_sync/_cli.py#L416) always persists the supplied
   overrides (or their defaults) into the manifest, while
   [add](../../src/context_sync/_cli.py#L440) preserves existing configuration.
   The unified command should adopt `add`'s preservation semantics as the
   no-override default and only persist traversal configuration when the
   operator explicitly supplies overrides.
5. Keep `refresh` as the explicit "update the current root set without changing
   it" command.
6. Keep `diff` as the explicit snapshot-wide read-only drift command for v1.
7. Use one consistent CLI placeholder and help term for the root argument
   across `sync` and `remove`. The recommended placeholder is `TICKET`,
   described as "Issue key or Linear URL of the ticket to track or remove."
8. Keep the underlying [ContextSync.remove_root()](../../src/context_sync/_sync.py#L1333)
   method unless the later API review identifies a stronger reason to rename
   the library surface too.

This yields a smaller, clearer public CLI:

- `sync <TICKET>`
- `refresh`
- `remove <TICKET>`
- `diff`

### Policy-adjacent clarity gaps

No blocking common-policy violation was found in the current flag set, but the
review did identify these consistency gaps:

1. The current CLI surface uses `root_ticket` for `sync` and `ticket_ref` for
   `add` / `remove-root`, even though all three positions accept a human-facing
   issue key or Linear URL. This is a terminology drift problem rather than a
   shared-policy violation. The CLI should use one consistent public placeholder
   such as `TICKET`.
2. The public library surface still exposes
   [ContextSync.sync(root_ticket_id=...)](../../src/context_sync/_sync.py#L290),
   while [M4-3](../implementation-plan.md#m4-3---rename-root-ticket-id-to-key)
   already acknowledges that `_id` is misleading for a human-facing key or URL.
3. The review conclusions above depend on source inspection and existing test
   artifacts. A live installed-environment CLI smoke path is still owned by
   [M5-2](../implementation-plan.md#m5-2---supported-public-runtime-validation-and-smoke-path),
   not by this review ticket.

### Output ergonomics

This section evaluates the operator-facing terminal surfaces: top-level and

#### Top-level help text

The top-level parser description at
[src/context_sync/_cli.py:390](../../src/context_sync/_cli.py#L390) reads

#### Subcommand help strings

| Command | Current help string | Assessment |
| --- | --- | --- |
| `sync` | "Full-snapshot sync from a root ticket." | Implementation-oriented. "Full-snapshot sync" exposes the internal rebuild-vs-incremental distinction that this review recommends hiding. A clearer string would describe the operator intent: "Track a ticket and update the snapshot." |
| `refresh` | "Incremental refresh of all tracked roots." | Implementation-oriented. "Incremental refresh" communicates an optimization strategy, not operator intent. A clearer string: "Update the snapshot from all tracked roots." |
| `add` | "Add a new root ticket and refresh the snapshot." | Accurate for the current surface but will be removed under the simplification. |
| `remove-root` | "Remove a root ticket and refresh the snapshot." | Accurate. Under the rename to `remove`, the string can stay similar. |
| `diff` | "Compare local snapshot to live Linear state (read-only)." | Good. Clearly communicates the read-only guarantee and the comparison direction. |

**Classification:** The `sync` and `refresh` help string improvements are
documentation-only text changes that can be applied without a plan amendment.
They should be addressed alongside the CLI simplification in
[FW-10](../future-work.md#fw-10-cli-simplification-amendment) since the command
set and help text are changing together.

#### Human-readable success/failure output

The success output formatting at
[src/context_sync/_cli.py:70](../../src/context_sync/_cli.py#L70) uses

[src/context_sync/_cli.py:98](../../src/context_sync/_cli.py#L98) uses

Error output from `ContextSyncError` subclasses uses human-readable messages

**Classification:** No changes needed for success/failure output text.
