  after each mutation to rule out stale reads. In that follow-up, comment
  creation still did **not** advance the parent issue `updatedAt`, and comment
  edit also did **not** advance the parent issue `updatedAt`.

  [docs/design/refresh-freshness-validation.md](../design/refresh-freshness-validation.md)
  as the durable outcome artifact, plus an ADR outcome pointer in
  [docs/adr.md](../adr.md#oq-1-refresh-freshness-validation-against-live-linear-behavior).
- Validation outcome:
  direct issue-field updates advanced `updatedAt`, but comment creation and
  comment edit did not advance the parent issue `updatedAt`, even after the
  longer follow-up wait. Issue-level `updated_at` is therefore insufficient
  for the v1 persisted main-ticket snapshot contract.
- Required amendment:
  `refresh` must move from a single issue-level freshness cursor to a
  composite per-ticket freshness signal before
  [M3-1](../implementation-plan.md#m3-1---incremental-refresh-and-quarantined-root-recovery)
  can begin.
- Scope boundary:
  this ticket intentionally stopped at recording the evidence and the minimum
  amendment shape. It did **not** design the replacement refresh contract
  because that governing-plan/design rewrite is outside the requested
  [M1-D1](../implementation-plan.md#m1-d1---refresh-freshness-validation-spike)
  deliverable and is instead being tracked as the draft plan amendment
  [docs/planning/change-requests/CR-1.md](../planning/change-requests/CR-1.md).
- Validation notes:
  this ticket modified docs only, so no repository-wide lint, format, or test
  commands were run. Validation consisted of cross-document consistency checks,
  local `linear-client` API inspection, live Linear schema introspection, and
  the disposable credentialed probe sequence described above.
