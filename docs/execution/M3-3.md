Key implementation decisions:

1. **Lock inspection**: `diff` calls `inspect_lock` + `is_lock_stale` without ever
   calling `acquire_lock`, `release_lock`, or unlinking the lock file. If the lock
   is demonstrably stale, diff proceeds with a log message. Otherwise it raises
   `DiffLockError` with an explanation of *why* lock refusal is intentional (rate-limit
   protection for the active writer).

2. **Changed-field detection**: Uses the existing batch metadata endpoints
   (`get_refresh_issue_metadata`, `get_refresh_comment_metadata`,
   `get_refresh_relation_metadata`) to compute remote cursor components without
   full-bundle fetches. Compares against the local `refresh_cursor` stored in
   frontmatter. Reports changed cursor components (`issue_updated_at`,
   `comments_signature`, `relations_signature`) plus `issue_key` for rename detection.

3. **Classification**: Follows the design contract exactly:
   - `current` — all cursor components match
   - `stale` — at least one cursor component differs
   - `missing_locally` — tracked in manifest but file absent on disk
   - `missing_remotely` — not visible in the current Linear view

4. **Module extraction**: `check_diff_lock` and `compute_changed_fields` are
   extracted into [src/context_sync/_diff.py](../../src/context_sync/_diff.py) to
   keep `_sync.py` within the 1 000 code-line guideline (953 code lines after
   extraction).

## Completion Summary

Implemented the `diff` method on `ContextSync`, replacing the M3-3 stub. The method:

1. Inspects the writer lock without acquiring/modifying it.
2. Loads the manifest to discover tracked tickets.
3. Reads local frontmatter for refresh cursor data.
4. Batch-fetches remote metadata via the three refresh metadata gateway methods.
5. Classifies each ticket as current, stale, missing_locally, or missing_remotely.
6. Returns a `DiffResult` with no files modified.

Integration tests cover all four required scenarios plus additional edge cases.

Validations performed: `ruff check`, `ruff format --check`, `pytest`.
