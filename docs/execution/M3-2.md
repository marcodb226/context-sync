### Key design decisions

1. **Reuse `_refresh_under_lock` rather than inline refresh logic.** The `add`
   and `remove_root` methods modify the manifest, persist it to disk, then
   delegate the whole-snapshot refresh to `_refresh_under_lock` which re-loads
   the manifest. This avoids duplicating the refresh pipeline.

2. **Always fetch during `add` to validate availability.** Even when alias
   resolution succeeds, `add` fetches the ticket to confirm it is still
   available and to validate workspace membership. The resolved UUID is used as
   the fetch key when available (avoids redundant key→UUID resolution).

3. **Zero-roots pruning.** When `_refresh_under_lock` encounters an empty root
   set, it now prunes all remaining tracked tickets and their files rather than
   returning a vacuous empty result. This is correct: if no roots exist,
   nothing is reachable, so all tracked tickets should be removed.
