## Work Log

- Opened the ticket execution artifact and verified the governing plan is the
  active [docs/implementation-plan.md](../implementation-plan.md) and that
  [M1-D2](../implementation-plan.md#m1-d2---linear-domain-coverage-audit-and-adapter-boundary)
  is a design ticket under the active execution model.
- Re-read the ticket notes in
  [docs/implementation-plan.md](../implementation-plan.md#m1-d2---linear-domain-coverage-audit-and-adapter-boundary),
  the Linear-boundary guidance in
  [docs/adr.md](../adr.md#31-foundation),
  the adapter rule and refresh-flow requirements in
  [docs/design/0-top-level-design.md](../design/0-top-level-design.md#11-linear-dependency-boundary)
  and
  [docs/design/0-top-level-design.md](../design/0-top-level-design.md#62-refresh-flow),
  the package summary in
  [docs/design/linear-client.md](../design/linear-client.md),
  and the accepted composite refresh contract recorded by
  [M1-D3](M1-D3.md).
- Confirmed that the repo-local `.venv` contains `linear-client` 1.0.0 and
  treated the installed package as the concrete audit target for this ticket.
- Inspected the installed public and near-public package surfaces used by the
  design: the `Linear` facade, `Issue`, `Team`, `Comment`, and `IssueLink`
  domain objects; the issue, comment, and attachment repositories; the
  GraphQL facade; the typed issue and user services; and the typed models used
  by those services.
- Confirmed that the domain layer already covers the single-ticket issue fetch
  path well enough for v1 root resolution and ordinary scalar issue reads:
  `linear.issue(...).fetch()` and `linear.gql.issues.get(...)` both expose the
  persisted issue scalar fields including `updated_at`.
- Confirmed that full comment-body and attachment-metadata reads are available
  through the domain layer via `await issue.comments` and
  `await issue.attachments`. Noted that the package already satisfies those
  domain calls by dropping to raw GraphQL internally, which is acceptable for
  this ticket because the fallback remains upstream-package-internal rather
  than scattered through `context-sync`.
- Confirmed that the installed comment surface is not sufficient for the
  [M1-D3](../implementation-plan.md#m1-d3---refresh-composite-freshness-contract-amendment)
  `comments_signature` contract. The audited domain and repository surfaces
  return comment body text and `createdAt`, but do not expose comment
  `updatedAt`, parent or root topology, thread `resolved`, or a deletion or
  tombstone signal.
- Confirmed that the installed package does not expose a read-only issue-
  relation listing API in the audited domain layer or typed GraphQL services.
  The package can create relations via `Issue.link(...)`, but traversal,
  rendered `relations`, and `relations_signature` all still lack a packaged
  read path.
- Confirmed that the installed package does not expose the batched by-id
  metadata reads that the v1 refresh design expects. The audited surfaces
  support single-ticket fetch and team-scoped search, but not a direct batch
  issue-metadata read for the tracked reachable set, nor packaged batched
  comment-metadata or relation-metadata reads.
- Chose the boundary outcome recorded in
  [docs/design/linear-domain-coverage-audit.md](../design/linear-domain-coverage-audit.md):
  use the domain layer by default for supported single-ticket reads, but allow
  one narrow local raw-GraphQL adapter boundary for relation reads and the
  composite refresh-metadata pass.
- Defined the allowed local fallback shape conservatively as five read helper
  categories:
  `get_issue_workspace_identity(issue_id)`,
  `get_ticket_relations(issue_ids)`,
  `get_refresh_issue_metadata(issue_ids)`,
  `get_refresh_comment_metadata(issue_ids)`,
  and `get_refresh_relation_metadata(issue_ids)`.
  Recorded that later tickets should call those helpers rather than using
  `linear.gql.query(...)` directly from traversal or refresh orchestration.
- Recorded that those local raw-GraphQL helpers are read-only adapter helpers:
  they must use `linear.gql.query(...)` for non-connection reads, use
  `linear.gql.paginate_connection(...)` or an equivalent exhaustive internal
  cursor loop for connection-backed reads, own pagination internally, and
  never use `linear.gql.gql(...)` or `linear.gql.mutate(...)`.
- Recorded the newly discovered upstream follow-up targets in
  [docs/design/linear-domain-coverage-audit.md](../design/linear-domain-coverage-audit.md):
  relation-read support, typed comment and thread freshness metadata,
  batched issue-metadata reads for arbitrary tracked-id sets, and a stable
  workspace-identity read surface.
- Recorded two risks explicitly rather than normalizing them away:
  the metadata-only comment freshness path may still be more expensive than
  intended, and this ticket could not confirm from the installed package
  whether comment-level `updatedAt` is available and advances on edit.
- After the independent Phase B review and accepted owner-response
  dispositions, tightened the audit further to resolve the accepted findings:
  added workspace-identity validation as an explicit required v1 read
  operation,
  removed team-scoped issue search from the required-operation matrix,
  expanded the allowed narrow-helper list to include a dedicated workspace-
  identity helper,
  specified the read-only and pagination contract for local raw-GraphQL
  helpers,
  rewrote the workspace-identity risk note as a confirmed required v1 gap
  rather than as a hypothetical future need,
  and recorded an explicit pre-
  [M3-1](../implementation-plan.md#m3-1---incremental-refresh-and-quarantined-root-recovery)
  gate for settling the accepted `comments_signature` input instead of leaving
  that decision to implementation time.
- Updated
  [docs/implementation-plan.md](../implementation-plan.md#m3-1---incremental-refresh-and-quarantined-root-recovery)
  to carry the same explicit pre-
  [M3-1](../implementation-plan.md#m3-1---incremental-refresh-and-quarantined-root-recovery)
  comment-freshness gate so downstream implementation work does not rediscover
  it ad hoc.

- Deliverables produced:
  the authoritative audit and boundary artifact in
  [docs/design/linear-domain-coverage-audit.md](../design/linear-domain-coverage-audit.md),
  a cross-link from
  [docs/design/linear-client.md](../design/linear-client.md)
  back to that audit,
  an added pre-
  [M3-1](../implementation-plan.md#m3-1---incremental-refresh-and-quarantined-root-recovery)
  comment-freshness gate in
  [docs/implementation-plan.md](../implementation-plan.md#m3-1---incremental-refresh-and-quarantined-root-recovery),
  and this execution record.
- Accepted boundary outcome:
  keep the domain layer as the default path for single-ticket issue fetches,
  full comment-body reads, and attachment metadata;
  reserve raw `linear.gql.*` usage for one narrow local adapter module that
  owns workspace-identity validation,
  relation reads,
  plus the three batched refresh-metadata reads required by
  the [M1-D3](../implementation-plan.md#m1-d3---refresh-composite-freshness-contract-amendment)
  contract;
  and require those helper implementations to stay read-only, own pagination
  internally, and use `query(...)` or `paginate_connection(...)` rather than
  `gql(...)` or `mutate(...)`.
- Recorded capability gaps:
  no packaged read-only relation listing path,
  no packaged typed comment or thread metadata path sufficient for
  `comments_signature`,
  no packaged batched by-id issue freshness read for the tracked reachable
  set,
  and no packaged stable workspace-identity read surface.
- Recorded risk and follow-up notes:
  the accepted metadata-only comment freshness path may still prove expensive,
  comment-level `updatedAt` behavior remains unconfirmed from the installed
  surface and is now an explicit pre-
  [M3-1](../implementation-plan.md#m3-1---incremental-refresh-and-quarantined-root-recovery)
  repository gate rather than an implementation-time choice,
  and workspace-identity validation is now recorded as a confirmed required v1
  package-surface gap rather than as a hypothetical later concern.
- Validation notes:
  this was a docs-only design ticket, so no repository-wide lint, format, or
  test commands were run. Validation consisted of manual package-surface
  inspection against the repo-local `linear-client` install plus manual
  cross-document consistency checks across the plan, ADR, top-level design,
  package summary, and the new audit artifact.
