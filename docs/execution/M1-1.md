## Coding Guidelines Compliance

Checked against
[docs/policies/common/coding-guidelines.md](../policies/common/coding-guidelines.md)
and
[docs/policies/common/coding-guidelines-python.md](../policies/common/coding-guidelines-python.md):

- **Documentation**: Every source file has a module docstring. All public
  classes, methods, and functions have docstrings with parameter/return
  documentation.
- **No magic numbers**: Named constants for all dimension defaults, ticket
  caps, concurrency limits, and format version.
- **Markdown links**: All cross-document references in this execution file use
  clickable Markdown links.
- **Type annotations**: All function signatures and class fields are annotated.
  `typing.Protocol` used for structural interface. Frozen dataclasses for all
  structured data containers.
- **Async**: All entry points are async. Semaphore-bounded concurrency created
  in the constructor.
- **Exception handling**: Specific exception types for each failure scenario.
  Custom exception hierarchy with documented semantics.
- **Testing**: Every new module has a corresponding test file. Both happy and
  unhappy paths tested. Fake gateway exercises all protocol methods.
- **DRY**: `resolve_dimensions` handles validation once. `make_issue` and
  `make_syncer` factories prevent fixture duplication.
- **Logging**: Logger created in `_sync.py`. Logging content deferred to
  implementation tickets that produce log-worthy events.
- **Fail loudly**: Invalid config (unknown dimensions, negative depths, zero
  limits) raises descriptive errors immediately.
- **No hardcoded secrets**: All credential handling deferred to `linear-client`
  and the existing `scripts/.linear_env.sh.sample` pattern.
- **Version metadata**: Canonical `__version__` in `version.py`, derived by
  `pyproject.toml` via `setuptools.dynamic`.
- **Stable-release changelog gate**: No changelog entry required — this is a
  pre-`1.0.0` repository (`0.1.0.dev0`).
