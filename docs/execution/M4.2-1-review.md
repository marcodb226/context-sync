## Residual Risks

- **Pyright `standard` vs `strict` gap.** The `standard` type-checking mode
  does not catch several `NewType` boundary crossings that would surface under
  `strict` (for example the `_ticket_ref.py` cross-type comparison and the
  `IssueKey(uid)` casts). This is by design for this ticket, but it means the
  type-safety net has known holes that only tighten when/if the mode is upgraded.

- **Timestamp fields remain untyped.** The `created_at` / `updated_at` fields
  across all gateway dataclasses are bare `str`. These fields cross multiple
  module boundaries (gateway, pipeline, signatures, renderer) and a
  mix-up between an RFC 3339 timestamp and another string value would not
  be caught.

## Testing Gaps

- No test specifically validates that `NewType` wrapping is preserved through
  round-trip operations (e.g. constructing a `SyncResult`, serializing to
  manifest YAML, reloading). This is inherently limited by `NewType` being
  erased at runtime, but property-based tests that verify type annotations
  match actual field values (via `get_type_hints`) would catch annotation drift.
